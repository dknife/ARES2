D.2

시뮬레이터 — 모놀리스 simulation.js의 모듈화

부록 A 기준부록 A(§A.2 simulation.js, 8장 미션 시뮬레이터)는 simulation.js를 three.js 기반 3D 시뮬레이션의 초기화·관리 단일 파일로 기술한다. 공개 API는 setupSimulation({workspace, onOpen, onClose}) 하나뿐이고, OLED_ICONS·TOPICS·buildSim·recolorLaunchpadAntenna 등은 거대 클로저 buildSim() 내부의 비공개 헬퍼였다(1차 진단의 “유형 A — 거대 클로저”).

통합 후「2차 Refactoring 계획」의 유형 A 실행으로, buildSim()에 묶여 있던 약 50개 상태와 서브시스템을 끌어올려 공유 컨텍스트(ctx) + 기능별 모듈 구조로 분해했다. simulation.js는 383줄의 얇은 오케스트레이터로 줄고, 기능은 Web/Sim_Parts/ 14개 모듈과 Web/Simulation/ 5개 래퍼로 분리됐다(setupSimulation() 외부 인터페이스는 그대로라 main.js는 변경 거의 없음).

모듈 (Web/Sim_Parts/)책임
context.js공유 상태(SimContext) + 씬·카메라·렌더러 구성, 서브시스템 조립
render.js / assets.js렌더 루프 / GLTF 로딩
movement·leds·gun·oled.js주행·LED·발사·디스플레이 동작
rocket·traffic·waves.js로켓 발사·신호등·음파 연출
audio·dispatch·topics·editor_controls.js음향·명령 디스패치·토픽 정의·객체 배치 편집

Web/Simulation/(Simulation_Main·Launcher·Rover·Traffic·AresRobot)은 위 라이브러리를 토픽별로 감싸 buildSim을 재노출하는 얇은 래퍼다. 머지에는 음파 생성(SpawnSoundWave) 보정 등 소규모 버그 수정과, 모바일 객체 배치(Move·Rotate·Scale)를 위한 editor_controls.js가 포함된다. 구 모놀리스 전문은 Web/simulation_backup.js로 보존된다.

D.3

피코 펌웨어 — 설정 체계와 명령 프로토콜 변경

부록 A 기준부록 A(2장 구성요소, 4장 설정 로딩, §A.1 system_config.py)는 설정 정본을 system.json(JSON)으로 기술한다 — CONFIG_FILE='system.json', 모듈 활성 플래그·핀 할당·보정값을 저장하고, SystemConfig가 입출력을 담당하며, 구형 calibration.json을 병합한다. 또한 9장은 설정이 펌웨어 system.json과 웹 localStorage에 이중으로 존재함을 지적했다.

통합 후JSON 설정을 폐지하고, 교사·어린이가 메모장으로 직접 고치는 텍스트 설정 ModelFactorySetting.txt로 대체했다.

  • 설정 파일 텍스트화 — system_config.py에 키=값(또는 키:값) 파서를 추가, 첫 부팅 시 설명이 담긴 기본 템플릿을 자동 생성하고, 저장 시 기존 주석·가이드라인을 보존하고 값만 갱신. system.json은 삭제.
  • 속도 기능 + 모터 속도 제한 — 서보·DC 이동 명령이 속도 인자를 받도록 형식을 확장하고, max_speed 기준으로 스케일링해 0~100%로 제한(5일차 모터 속도 제한 코드). process_data.py는 653→713줄.
  • 커스터마이징 확장 — 블록코딩 탭 한글 이름(GET_NAMES), 테마 모델(로버/발사대), 모듈 토글, GP 핀 번호, 서보 좌우 보정치를 모두 설정 파일에서 제어.
명령 형식의미 (신규/확장)
SERVO_방향,속도서보 휠 연속 이동 + 속도(0~100%)
SERVO_t방향,초,속도시간 제한 서보 이동 + 속도
DC_방향,속도 / DC_t방향,초,속도DC 모터 연속/시간 제한 + 속도
GET_NAMES블록코딩 탭 한글 이름 조회 (신규)
⚠ 주의 — 계약 문서 갱신 필요
명령 프로토콜이 바뀌었으므로(예: SERVO_방향,속도, GET_NAMES) 부록 A가 사실상의 계약 문서로 지목한 API_DOCUMENTATION.md와 펌웨어·웹 양측을 짝으로 갱신해야 한다. 기존 기기의 system.json 값은 새 ModelFactorySetting.txt로 자동 이전되지 않는다(기본 템플릿 초기화).
D.4

웹 인터페이스 — 라우팅 정리와 블록코딩·BLE 연동

부록 A 기준부록 A(§A.2 main.js)는 main.js를 라우팅·뷰 전환·Blockly 초기화·이벤트·버튼 콜백을 대부분 내부 함수로 담당하는 진입점으로 기술한다(1차 진단에서 “한 함수가 미션/저장/예제/AI/대시보드를 전부 처리”로 지목). 7장은 새 블록 추가 시 blocklyconfig.js·툴박스·commandexecutor.js 세 곳을 함께 고쳐야 함을 강조했다.

통합 후

  • UI 보조 분리 — updateBlockCodingButtonUI·setupLogToggle·setupContentToggle를 ui.js(신규 97줄)로 추출(main.js는 1,051→1,473줄로, 모바일 동선 로직 추가분 포함).
  • 모바일 하단 내비 + 버튼 동선 — 개요·차시·미션·AI·로그 하단 고정 바(safe-area-inset 대응)와 상단 버튼(개요·블록코딩·점검) 이동 버그 수정. styles.css는 모바일·반응형으로 +455줄.
  • 명령 프로토콜 연동 — 속도 기능·GET_NAMES에 맞춰 blocklyconfig.js·bluetooth.js·commandexecutor.js를 함께 갱신(부록 A가 지적한 “세 곳 동시 수정” 지점이 그대로 반영됨).
D.5

진입점·자산 — 페이지 구성과 3D 모델 경량화

부록 A 기준부록 A는 진입점으로 main.html(블록 편집기)·dashboard.html(점검)을, 모바일 미리보기로 mobile-preview.js를 기술한다.

통합 후공개 랜딩(루트 index.html)·개발 랜딩(dev.html)과 메인 앱(Web/index.html)으로 페이지 구성이 확장됐고(main.html/styles.css에 모바일 내비·3D 히어로 추가), 3D 모델 ares_robot.glb는 7.21 MB → 0.58 MB로 경량화(약 92% 감소), 우주인 모델 Astronaut.glb가 추가됐다. 이 코드 변경은 개발 문서(CLAUDE.md “Web UI Module Structure”)에도 반영했다.

중간 점검 결론
1·2차 Refactoring 계획의 핵심 항목 — 유형 A 거대 클로저 분할(시뮬레이터)과 피코 설정·명령 체계 정비 — 이 통합 머지로 실제 코드에 반영됐다. 남은 과제는 부록 A가 짚은 설정 이중화(system.json ↔ localStorage)의 단일화 마무리와, 바뀐 명령 프로토콜에 맞춘 API_DOCUMENTATION.md 갱신이다.
