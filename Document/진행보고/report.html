2025년 겨울·준비단계 구현 내용 — 배포된 웹 서비스

dknife.github.io/ARES2

6월 현장 착수 이전 2025년 겨울부터 준비단계에 구현해 GitHub Pages로 배포한 ARES 웹 서비스다. 이후 1~11일차 현장 개발은 이 준비단계 결과물을 기반으로 진행됐다. 설치 없이 브라우저만으로 학습 콘텐츠 열람 · 블록 코딩 · 3D 시뮬레이션 · AI 도움 · 기기 모니터링이 가능하다. 블록 1개가 명령 문자열로 변환돼 BLE로 로버에 전달되는 전체 경로를, 실물 기기 없이도 시뮬레이션으로 체험할 수 있다.

🚀 주요 기능 — 학습 관제실부터 3D 시뮬레이션까지

랜딩 페이지에서 ‘서비스 시작’으로 학습 관제실에 진입하면 12차시 × 4미션 = 48개 미션의 커리큘럼(‘화성 탐사선 코딩 여정’)이 차시 카드와 표로 안내된다. 미션을 고르면 Google Blockly 기반 블록 코딩 화면이 열리고, ‘시뮬레이션’으로 탐사로봇 ‘알비’가 3D로 등장해 작성한 블록의 동작을 미리 보여 준다.

랜딩 · 학습 관제실 블록 코딩(Blockly) WebGL 3D 시뮬레이션 시뮬 모델 4종 AI 도움 점검 대시보드

아래 화면은 dknife.github.io/ARES2를 데스크톱 브라우저로 구동해 캡처했다. 블루투스(BLE)는 사용자 제스처·실기기가 필요해 연결 상태는 ‘연결 끊김’으로 표시되며, 블록 코딩·3D 시뮬레이션·학습 콘텐츠는 기기 없이도 동작한다.

🎮 시뮬레이션 모델 4종 — 드롭다운 전환

시뮬레이션 화면 좌측 상단의 모델 선택 드롭다운으로 4종의 모델을 전환할 수 있다. 같은 블록 코드라도 선택한 모델에 따라 서로 다른 3D 장면·동작이 연출되어, 미션 맥락(로봇·신호등·발사·주행)에 맞는 학습이 가능하다. 네 모델을 차례로 선택해 각각 다른 시뮬레이션이 이뤄짐을 확인했다.

2026.06.23 (월) · 10:00–17:00 KST

📄 주요 업무 — 코드 분석 종합 보고서 작성

ARES 저장소(github.com/dknife/ARES2) 전체를 코드 수준에서 분석한 「ARES 프로젝트 코드 분석 종합 보고서」(전 14장)를 작성했다. 웹·펌웨어·AI 세 런타임 도메인이 개행 구분 텍스트 명령이라는 단일 계약으로 느슨하게 결합되는 구조를 중심으로, 하드웨어부터 향후 연구 방향까지 망라했다.

개요·구성요소 하드웨어·핀 배치 피코 펌웨어(CommandProcessor) 웹 서비스 계층 Web Bluetooth 블록 코딩 구현 WebGL 3D 시뮬레이션 코드 연계 구조 네이티브 앱 패키징 개선 계획·연구 주제
  • 대상 시스템 — 초등학생용 블록 코딩 화성 탐사 로버 ‘알비’: Blockly 웹앱 + 라즈베리파이 피코(MicroPython) 펌웨어 + AI 보조 모듈.
  • 핵심 발견 — 웹과 펌웨어는 코드를 공유하지 않고 텍스트 명령 프로토콜만으로 통신해 독립 개발·교체가 쉬운 반면, 프로토콜 변경 시 양측 동시 반영 의존이 생긴다.
  • 범위 — 블록 1개가 ‘블록 → 명령 문자열 → BLE → 펌웨어 핸들러 → 하드웨어’로 도달하는 전 경로를 코드로 추적, 오프라인 file:// 단독 실행 구조까지 분석.

🧩 주요 업무 — 1차 Refactoring 계획 수립

지나치게 큰 함수·파일로 분석이 어려운 소스를 기능 단위로 분할하기 위한 「1차 Refactoring 계획」을 수립했다. 동작 보존(behavior-preserving)을 원칙으로, 비대 지점을 네 가지 문제 유형으로 분류하고 6단계 로드맵을 도출했다.

A · 거대 클로저 B · 거대 디스패처 C · 데이터 덩어리 D · 인라인 자산
  • 유형 A — 거대 클로저(simulation.js 2333줄)를 공유 컨텍스트(ctx) + 서브시스템 14개로 분할.
  • 유형 B — 거대 디스패처(process_data.py 170줄 if/elif)를 도메인 핸들러 + dict 디스패치로 전환.
  • 유형 C — 한 배열·만능 함수(blocklyconfig.js)를 카테고리별 파일로 분리.
  • 유형 D — 인라인 자산(dashboard.html 828줄)의 CSS·JS를 외부 파일로 추출.
  • 로드맵 — 위험 낮은 D·C부터 시작해 가장 어려운 A(simulation.js)를 마지막에 두는 6단계 점진 분할.

1차시 운영 점검 · 2차 Refactoring 계획

2026.06.26 (목) · 10:00–17:00 KST

수업 적용을 앞두고 1차시 운영을 점검했다. 조립된 로버·컨트롤러의 제품 동작과 교구 학습 진도를 확인하고, 팀이 함께 웹 인터페이스와 WebGL 3D 시뮬레이션을 점검해 「2차 Refactoring 계획」을 수립했다. 또한 최종 제품의 배선·외형 조립이 용이하도록 모듈형 패널로 분해·결합되는 조립형 모델 시작품을 제작해 조립성을 검증했다.

✅ 주요 업무 — 1차시 운영 점검 (제품 동작 · 교구 학습)

ARES 키트의 실제 수업 적용을 앞두고 1차시 운영을 점검했다. 조립된 로버와 컨트롤러의 동작을 실물로 확인하고, 화면의 학습 콘텐츠로 차시별 교구 학습 진도를 점검했다.

제품 동작 확인 컨트롤러 배선 방향 표시 교구 교구 학습 진도 1차시 운영
  • 제품 동작 확인 — 4륜 로버와 3D 프린팅 컨트롤러 케이스(입출력 핀이 라벨링된 보드·전원 LED)를 연결하고, 방향 표시 디스크 교구와 함께 실제 동작을 점검.
  • 교구 학습 진도 점검 — 모니터에 ARES 학습 콘텐츠(‘화성 탐사선 코딩 여정’)를 띄워 차시별 학습 진도를 점검하고, 발사대 모형과 함께 수업 흐름을 확인.

🧭 주요 업무 — 웹 인터페이스·3D 시뮬레이션 점검 & 2차 Refactoring 계획 수립

웹 인터페이스 점검 WebGL 3D 시뮬레이션 사용성 리뷰 2차 Refactoring 계획 조립 가능한 시작품 제작 완료
  • 웹 인터페이스 점검 — 컴퓨터실에서 팀원들이 함께 ARES 웹 페이지를 띄워 화면 구성과 동작을 점검.

🔍 문제 점검 — 전체 코드 리뷰 (웹·펌웨어·시뮬레이터·통신 프로토콜)

네 영역을 병렬로 정밀 점검했다: ① 웹 코어(블록 정의↔툴박스↔실행기 연결·BLE 송수신), ② 3D 시뮬레이터(모듈 import/export·에셋 경로·디스패치), ③ Pico 펌웨어(명령 디스패치·모듈 가드·MicroPython 제약), ④ 세 계층이 주고받는 명령 문자열 프로토콜의 상호 정합성. 블록↔툴박스 매핑, 파일명 대소문자, GLTF 경로, BLE 20바이트 청킹, fire-and-forget 명령 목록의 웹↔펌웨어 동기화 등은 이상 없음을 확인했고, 아래 4건을 치명(HIGH)으로 분류했다.

4개 영역 병렬 점검 HIGH 4건 MEDIUM·LOW 20여 건 연결·정합성 다수 확인
  • ① 자석 센서 표시 반전 (dashboard.html) — 펌웨어(magsensor.py)는 1=감지로 보내는데 대시보드는 mag===0일 때 "감지됨"을 표시. 자석이 없으면 "감지됨", 있으면 "정상"으로 나오는 완전 반전 버그.
  • ② 설정 파일 오타 한 줄로 로버 부팅 불가 (system_config.py) — is_module_enabled()가 int()를 보호 없이 호출. 사용자가 설정 파일에 enable_wheel=yes처럼 쓰면 import 시점(hardware.py가 호출)에 예외가 발생해 main.py가 아예 시작하지 못함.
  • ③ 5초 초과 명령의 BLE 응답 어긋남 (bluetooth.js) — 시간지정 이동(예: 10초 전진)은 펌웨어가 동작 완료 후에야 ack를 보내는데 웹 타임아웃은 고정 5초. 타임아웃 뒤 도착한 늦은 ack가 다음 명령의 응답 슬롯을 오염시켜, 이후 "거리 측정" 값이 어긋나는 등 응답 스트림이 한 칸씩 밀림.
  • ④ 비상정지 지연 — 이중 차단: 웹은 STOP_ALL이 전송 큐에 갇혀 앞 명령의 응답/타임아웃까지 대기하고, 펌웨어는 시간지정 이동을 상한 없는 blocking sleep으로 처리해 그동안 어떤 명령도 못 받음(tFORWARD,9999면 수 시간). 어린이용 로봇에서 실질적 안전 결함.
  • MEDIUM·LOW (후속 수정 예정) — Web/Simulation/ 폴더가 실제로는 어디서도 import되지 않는 죽은 코드(문서와 불일치), SET_PIN이 저장만 되고 미적용, 블록 실행기의 else-if·반환값 함수 본문 누락, 대시보드 연결 상태 5초 후 오표시, 프레임레이트 종속 애니메이션 등.

📝 수정 계획 — 치명(HIGH) 4건

발견 당일 4건의 수정 계획을 수립했다. 원칙은 (a) 근본 원인 제거, (b) 웹·펌웨어 양쪽 방어, (c) 자동화 검증 동반이다.

  • ① 자석 반전 — 대시보드 판정 조건을 펌웨어 규약(1=감지)에 맞게 뒤집는다(한 줄).
  • ② 부팅 안전화 — is_module_enabled()를 try/except로 감싸 잘못된 값이면 크래시 대신 해당 모듈만 비활성으로 강등하고 시리얼 로그로 원인을 알린다.
  • ③ 응답 동기화(2단 방어) — (a) 명령 문자열에서 예상 소요시간을 계산해 타임아웃을 동적으로 연장(SERVO_t*/DC_t*/SLEEP은 초 인자, BATCH는 하위 명령 합산), (b) 값 반환 명령(DISTANCE→DIST: 등)은 기대 응답 접두어를 검증해 무관한 늦은 ack를 버리고 계속 대기. 부수로 대시보드 저장 타임아웃 기본값(20000ms)을 5000ms로 일치.
  • ④ 비상정지 즉시화 — 웹: 비상정지 시 대기 중 응답 promise를 즉시 취소해 전송 큐를 비우고 STOP_ALL을 바로 내보냄. 펌웨어: blocking sleep을 50ms 슬라이스로 바꾸고 매 슬라이스 UART를 폴링해 STOP_ALL 도착 시 즉시 정지. 시간 상한 60초를 웹·펌웨어 양쪽에 동기 적용.

🔧 수정 내용 — 적용 결과 및 자동화 검증

계획대로 8개 파일에서 +224 / −34 라인을 수정했다(웹 5·펌웨어 3). 핵심 구현은 펌웨어의 중단 가능 sleep과 웹의 동적 타임아웃·응답 짝 검증이다.

  • 🔩 펌웨어 (process_data.py·main.py·system_config.py) — _interruptible_sleep() 도입(시간지정 휠·DC·SLEEP·BATCH 부저 대기 5곳 교체), main.py의 _poll_abort()가 동작 중 UART를 폴링해 STOP_ALL만 선별 소비, BATCH는 중단 시 남은 하위 명령을 건너뜀. MAX_TIMED_SEC=60 클램프.

1~9일차의 현장 기록을 2주차 중간 시점에서 종합한다. 이 기간 ARES는 웹 UI·3D 시뮬레이션·피코 펌웨어의 세 런타임을 병행 개발하면서, 팀원 브랜치의 main 통합 머지로 흩어진 작업을 하나의 서비스로 합치고, 전체 코드 정밀 점검(치명 4건·중간 5건 수정)으로 안정성을 끌어올렸다. 구조 면에서는 부록 A 기준선의 모놀리식 simulation.js(약 2,300줄)를 공유 컨텍스트 + 14개 서브시스템으로 분할(유형 A)했고, 디자인 면에서는 코리아사이언스 멘토링을 거쳐 화성 탐사 콘셉트의 컬러·모듈 디자인 전략을 확정하고 이를 앱 전 화면 UI 시안 → 실제 웹 서비스와 실물 교구(슬라이드형 로버·화성발사대)로 완성했다. 아래 다섯 절에서 (1) 개발 주요 요소, (2) 부록 A 대비 SW 구조 개선, (3) 코드 분할·연계성, (4) 교구 시제품 디자인, (5) UI/UX 개선을 정리한다.

🧱 SW 구조 개선 — 부록 A 코드의 구조 변화 · 내용 변화 요약

부록 A는 프로젝트 1일차에 작성한 「ARES 코드 분석 종합 보고서」(전 14장 + 소스별 API Reference)로, 통합 머지 이전 main(기준 커밋 6f49e27, 기준 시점 2026-06-21)의 코드 상태를 서술한 기준선이다. 그 기준선 대비 2주간의 구조·내용 변화를 요약한다(전문 대조는 부록 D).

구조 변화 — 모놀리식에서 모듈 분할로

  • 시뮬레이터 — simulation.js 단일 파일 약 2,300줄(buildSim() 거대 클로저) → simulation.js 383줄(얇은 오케스트레이터) + Sim_Parts/ 14모듈 + Simulation/ 5래퍼. 원본은 simulation_backup.js로 보존.
  • 웹 코어 — main.js 약 1,051줄 → ui.js(신규 97줄) 추출(버튼 UI·로그/콘텐츠 토글). 나머지 코어(bluetooth·commandexecutor 등)는 응집도 높아 유지.
  • 피코 설정 — 정본 system.json(JSON) → 텍스트 ModelFactorySetting.txt(키=값, 주석·가이드 보존). 싱글턴 sys_config·모듈 가드 패턴은 유지.

내용 변화 — 안정성·프로토콜·성능 (통합 머지로 Web/·Pico/·AI/에서 35개 파일 +5,152/−2,563 라인 변경, 메시 자산·백업 제외)

치명 4건 수정 중단 가능 sleep · 동적 타임아웃 BLE ~250ms → ~40ms 프로토콜 확장(속도·GET_NAMES) glb 7.21MB → 0.58MB
  • 치명(HIGH) 4건 수정 — 자석 표시 반전·설정 오타 부팅 불가·BLE 응답 어긋남·비상정지 지연. 근본 원인 제거 + 웹·펌웨어 양쪽 방어 + 자동화 검증 원칙 적용.
  • 중단 가능 sleep · 동적 타임아웃 — 펌웨어가 시간지정 대기를 50ms 슬라이스로 쪼개 매번 비상정지를 폴링(상한 MAX_TIMED_SEC=60), 웹은 _estimateBlockingMs()로 타임아웃을 명령별로 산출. 실측 10초 이동이 0.36초에 중단.
  • BLE 파이프라인 최적화 — fire-and-forget 정책(FIRE_AND_FORGET_HEADS ↔ NO_RESPONSE_CMDS)과 UART 라인 폴링으로 명령 지연 ~250ms → ~40ms, 처리량 ~16 → 100+ cmd/s.
  • 프로토콜 확장 — 속도 인자(SERVO_방향,속도·DC_t방향,초,속도), 한글 이름 조회 GET_NAMES 신설, 블록 인터프리터의 else-if·반환값 함수 실행.
  • 자산 최적화 — ares_robot.glb 7.21MB → 0.58MB(~92% 감소), Astronaut.glb 추가(meshopt + WebP).

🔗 SW 코드 분할 내역 및 상호 연계성 분석

분할의 핵심은 시뮬레이터이며, 부록 A의 유형 A(거대 클로저) 처방대로 공유 컨텍스트 + 기능 서브시스템으로 재편했다.

시뮬레이터 분할 내역 (1파일 → 14+5 모듈)

  • 허브 — context.js(SimContext: 공유 상태 + 씬·카메라·렌더러, buildSim 조립), topics.js(TOPICS·OLED_ICONS·팔레트).
  • 표현 — render.js(렌더 루프), assets.js(GLTF 로딩), oled.js, leds.js, gun.js, waves.js(부저 음파), audio.js(사운드 합성).
  • 동작 — movement.js(주행·레이더·충돌·거리센서), rocket.js(로켓 발사·카메라 추적), traffic.js(신호등·가위바위보).
  • 연동 · 편집 — dispatch.js(CommandDispatcher: 명령 문자열 → 시각 효과), editor_controls.js(모바일 객체 Move/Rotate/Scale 배치 편집, 신규).
  • 토픽 래퍼 — Simulation/: Simulation_Main·Launcher·Rover·Traffic·AresRobot(각 Sim_Parts를 토픽별로 확장).
현행 배선 사실 확인 — 현재 main 배선은 simulation.js → Sim_Parts/context.js 직접이며, Simulation/의 5개 래퍼는 현재 어디에서도 import되지 않는 죽은 코드이다(팀원 브랜치 Simulation_LeeMinhyuck가 Simulation_Main 경유 재배선 진행 중 — 머지 리뷰 대상). 문서(CLAUDE.md)의 "Simulation_Main.js로 배선" 설명은 후속 반영 예정으로 본다.

상호 연계성 (의존 그래프)

  • 웹 — 엔트리 main.js가 state·elements·logger·bluetooth·blocklyconfig·commandexecutor·simulation·ui·ai_helper를 import. commandexecutor.js는 bluetooth.js(송신)와 blocklyconfig.js(블록 정의)에 의존하고, bluetooth.js는 하위 의존이 없는 하드웨어 통신 말단. 시뮬레이터 허브 context.js가 ctx(=this)를 각 서브시스템에 주입.
  • 피코 — main.py(AresRover) → process_data.py(CommandProcessor, 유형 B 디스패처) → hardware.py(싱글턴 robot) → 각 모듈. hardware·process_data는 system_config.py(싱글턴 sys_config)의 모듈 활성 여부로 가드.
  • 데이터 흐름 — 블록 → commandexecutor가 명령 문자열 생성 → bluetooth → BLE → 피코 UART(9600) → CommandProcessor → 싱글턴 robot의 모듈 구동 → 값 반환/blocking 명령만 20바이트 청크 응답. 시뮬레이션 경로는 동일 명령을 simSink → dispatch.js가 시각 효과로 라우팅(하드웨어 없이 동작).
  • 횡단 결합점(동기화 필수) — ① 블록 3중 결합(blocklyconfig의 type ↔ commandexecutor의 case ↔ ai_helper), ② 문자열 프로토콜(컴파일 검증 불가 — API_DOCUMENTATION.md가 유일 계약, 웹·펌웨어 짝 갱신 필요), ③ 동기 상수쌍(MAX_TIMED_SEC=60, fire-and-forget 목록, 수신 버퍼 bytes화).

🔘 서비스 앱 — 맥락 인지형 중앙 버튼 UI

서비스 앱 하단 중앙 버튼을 맥락 인지형(context-aware)으로 구현했다. 같은 자리의 버튼 하나가 사용자의 상황(화면 모드·연결 상태·실행 여부)에 따라 색·아이콘·라벨·동작을 스스로 바꿔 그때 필요한 조작만 노출한다 — 연결이 안 됐으면 실행 대신 연결을, 시뮬레이션은 연결 없이 모의실행을 안내한다.

상황별 자동 전환 신호연결↔연결됨 실행↔비상정지 모의실행↔실험중단 연결 없이 시뮬
  • 미연결 — 빨강 ‘신호연결’(링크). 누르면 탐사선 연결 시도.
  • 연결됨(비코딩) — 초록 ‘연결됨’.
  • 코딩·연결 — 골드(#E6AD00) ‘실행’(재생). 블록을 로버로 전송. 미연결이면 ‘신호연결’로 표시(실행 불가).
  • 코딩 실행 중 — #B94A45 ‘비상정지’(정지). 누르면 즉시 정지(STOP_ALL).
  • 시뮬레이션 — 연결 없이 블루(#3F78A8) ‘모의실행’(재생).
  • 시뮬 실행 중 — #B94A45 ‘실험중단’(정지). 누르면 시뮬 중지.

💿 FinalBuild.md와 file://을 통한 오프라인 서비스

인터넷·서버 없는 교실·USB 배포를 위해 Web/을 file://로 더블클릭만 해도 동작하는 자족 오프라인 빌드(Build/)를 완성했다. 신규 씬 메시 인라인 누락을 바로잡고, 히어로 로봇 임베드(AlbiRobot.embed.js)를 교정했으며, 컷씬 로켓·유영 우주인은 fetch shim으로, 화성 배경·설정 아이콘은 data URI로 치환해 file:// CORS 차단을 해소했다. 학교 내부망 배포를 위해 Build/를 git 추적으로 전환했다.
