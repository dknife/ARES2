배포된 웹 서비스 주요 기능

dknife.github.io/ARES2

GitHub Pages로 배포된 ARES 웹 서비스를 직접 구동하며 주요 기능을 점검했다. 설치 없이 브라우저만으로 학습 콘텐츠 열람 · 블록 코딩 · 3D 시뮬레이션 · AI 도움 · 기기 모니터링이 가능하다. 블록 1개가 명령 문자열로 변환돼 BLE로 로버에 전달되는 전체 경로를, 실물 기기 없이도 시뮬레이션으로 체험할 수 있다.

🚀 주요 기능 — 학습 관제실부터 3D 시뮬레이션까지

랜딩 페이지에서 ‘서비스 시작’으로 학습 관제실에 진입하면 12차시 × 4미션 = 48개 미션의 커리큘럼(‘화성 탐사선 코딩 여정’)이 차시 카드와 표로 안내된다. 미션을 고르면 Google Blockly 기반 블록 코딩 화면이 열리고, ‘시뮬레이션’으로 탐사로봇 ‘알비’가 3D로 등장해 작성한 블록의 동작을 미리 보여 준다.

랜딩 · 학습 관제실 블록 코딩(Blockly) WebGL 3D 시뮬레이션 시뮬 모델 4종 AI 도움 점검 대시보드
웹 서비스 열기 ↗ 학습 관제실(로컬) ↗

아래 화면은 dknife.github.io/ARES2를 데스크톱 브라우저로 구동해 캡처했다. 블루투스(BLE)는 사용자 제스처·실기기가 필요해 연결 상태는 ‘연결 끊김’으로 표시되며, 블록 코딩·3D 시뮬레이션·학습 콘텐츠는 기기 없이도 동작한다.

🎮 시뮬레이션 모델 4종 — 드롭다운 전환

시뮬레이션 화면 좌측 상단의 모델 선택 드롭다운으로 4종의 모델을 전환할 수 있다. 같은 블록 코드라도 선택한 모델에 따라 서로 다른 3D 장면·동작이 연출되어, 미션 맥락(로봇·신호등·발사·주행)에 맞는 학습이 가능하다. 네 모델을 차례로 선택해 각각 다른 시뮬레이션이 이뤄짐을 확인했다.

DAY 1

컴퓨터실 개발 착수

2026.06.23 (월) · 10:00–17:00 KST

컴퓨터실에서 개발에 착수했다. 로버 하드웨어(섀시·모터 컨트롤러 보드)를 조립하고, 발사대·로켓·LED 모형을 제작했으며, 블록 코딩과 펌웨어 작업을 병행했다.

📄 주요 업무 — 코드 분석 종합 보고서 작성

ARES 저장소(github.com/dknife/ARES2) 전체를 코드 수준에서 분석한 「ARES 프로젝트 코드 분석 종합 보고서」(전 14장)를 작성했다. 웹·펌웨어·AI 세 런타임 도메인이 개행 구분 텍스트 명령이라는 단일 계약으로 느슨하게 결합되는 구조를 중심으로, 하드웨어부터 향후 연구 방향까지 망라했다.

개요·구성요소 하드웨어·핀 배치 피코 펌웨어(CommandProcessor) 웹 서비스 계층 Web Bluetooth 블록 코딩 구현 WebGL 3D 시뮬레이션 코드 연계 구조 네이티브 앱 패키징 개선 계획·연구 주제
  • 대상 시스템 — 초등학생용 블록 코딩 화성 탐사 로버 ‘알비’: Blockly 웹앱 + 라즈베리파이 피코(MicroPython) 펌웨어 + AI 보조 모듈.
  • 핵심 발견 — 웹과 펌웨어는 코드를 공유하지 않고 텍스트 명령 프로토콜만으로 통신해 독립 개발·교체가 쉬운 반면, 프로토콜 변경 시 양측 동시 반영 의존이 생긴다.
  • 범위 — 블록 1개가 ‘블록 → 명령 문자열 → BLE → 펌웨어 핸들러 → 하드웨어’로 도달하는 전 경로를 코드로 추적, 오프라인 file:// 단독 실행 구조까지 분석.

🏭 주요 업무 — 제작 협력업체 방문 & 웹 서비스 시연

키트 양산을 담당하는 제작 협력업체(D‑RISE)의 레이저 커팅 공장을 방문해 부품 생산 공정을 현장에서 점검하고, 완성된 ARES 블록 코딩 웹 서비스를 관계자에게 시연·리뷰했다.

레이저 커팅 부품 생산 공정 점검 품질·물류 협의 웹 서비스 시연 앱·부품 통합 검토
  • 생산 공정 견학 — GPL.TECH 레이저 커팅기로 가공되는 목재 키트 부품의 절단·정밀도·후처리 공정을 확인.
  • 부품·재고 점검 — 창고의 키트 부품 재고·포장 상태를 보고 양산·물류 일정을 협의.
  • 웹 서비스 시연 — 협력업체 관계자에게 ARES 웹 서비스(“블록 코딩으로 화성을 탐사하다”)를 시연하고 피드백을 수렴.
  • 통합 검토 — 레이저 커팅 부품과 태블릿의 블록 코딩 앱을 함께 놓고 실제 사용 흐름을 점검.

DAY 3

개발 지속 · 디자인 프로토타입 제작

2026.06.25 (수) · 10:00–17:00 KST

컴퓨터실로 복귀해 개발을 이어갔다. 통합·검증 작업과 함께 1차 리팩토링 계획을 수립하고, 자체 설계 디자인을 3D 프린터로 출력해 프로토타입을 점검하며 추가 디자인 계획을 세웠다.

🧩 주요 업무 — 1차 Refactoring 계획 수립

지나치게 큰 함수·파일로 분석이 어려운 소스를 기능 단위로 분할하기 위한 「1차 Refactoring 계획」을 수립했다. 동작 보존(behavior-preserving)을 원칙으로, 비대 지점을 네 가지 문제 유형으로 분류하고 6단계 로드맵을 도출했다.

A · 거대 클로저 B · 거대 디스패처 C · 데이터 덩어리 D · 인라인 자산
  • 유형 A — 거대 클로저(simulation.js 2333줄)를 공유 컨텍스트(ctx) + 서브시스템 14개로 분할.
  • 유형 B — 거대 디스패처(process_data.py 170줄 if/elif)를 도메인 핸들러 + dict 디스패치로 전환.
  • 유형 C — 한 배열·만능 함수(blocklyconfig.js)를 카테고리별 파일로 분리.
  • 유형 D — 인라인 자산(dashboard.html 828줄)의 CSS·JS를 외부 파일로 추출.
  • 로드맵 — 위험 낮은 D·C부터 시작해 가장 어려운 A(simulation.js)를 마지막에 두는 6단계 점진 분할.

🛠️ 주요 업무 — 디자인 프로토타입 출력 & 추가 디자인 계획

자체 설계한 디자인을 3D 프린터로 출력해 실물 프로토타입의 형상·조립성을 검증하고, Autodesk 3ds Max로 후속 디자인을 모델링하며 추가 디자인 계획을 수립했다.

3D 프린팅 출력 프로토타입 조립 점검 3ds Max 모델링 추가 디자인 계획
  • 프로토타입 출력 — 부저 스피커 베이스(‘코리아사이언스’ 각인)·로켓 노즈콘·LED 마운트 등 자체 제작 부품을 3D 프린팅하고, 컨트롤러·모터와 함께 배치해 조립성을 점검.
  • 추가 디자인 모델링 — Autodesk 3ds Max로 로버·발사대 구조의 후속 디자인을 3D 모델링.
  • 디자인 계획 수립 — 출력물·스케치·3D 모델을 대조하며 다음 단계 디자인 방향과 보완 사항을 정리.
DAY 4

1차시 운영 점검 · 2차 Refactoring 계획

2026.06.26 (목) · 10:00–17:00 KST

수업 적용을 앞두고 1차시 운영을 점검했다. 조립된 로버·컨트롤러의 제품 동작과 교구 학습 진도를 확인하고, 팀이 함께 웹 인터페이스와 WebGL 3D 시뮬레이션을 점검해 「2차 Refactoring 계획」을 수립했다. 또한 최종 제품의 배선·외형 조립이 용이하도록 모듈형 패널로 분해·결합되는 조립형 모델 시작품을 제작해 조립성을 검증했다.

✅ 주요 업무 — 1차시 운영 점검 (제품 동작 · 교구 학습)

ARES 키트의 실제 수업 적용을 앞두고 1차시 운영을 점검했다. 조립된 로버와 컨트롤러의 동작을 실물로 확인하고, 화면의 학습 콘텐츠로 차시별 교구 학습 진도를 점검했다.

제품 동작 확인 컨트롤러 배선 방향 표시 교구 교구 학습 진도 1차시 운영
  • 제품 동작 확인 — 4륜 로버와 3D 프린팅 컨트롤러 케이스(입출력 핀이 라벨링된 보드·전원 LED)를 연결하고, 방향 표시 디스크 교구와 함께 실제 동작을 점검.
  • 교구 학습 진도 점검 — 모니터에 ARES 학습 콘텐츠(‘화성 탐사선 코딩 여정’)를 띄워 차시별 학습 진도를 점검하고, 발사대 모형과 함께 수업 흐름을 확인.

🧭 주요 업무 — 웹 인터페이스·3D 시뮬레이션 점검 & 2차 Refactoring 계획 수립

ARES 웹 서비스의 인터페이스와 WebGL 3D 시뮬레이션을 팀이 함께 점검하고, 그 결과를 바탕으로 시뮬레이터·피코+블록코딩·웹 인터페이스 세 트랙의 「2차 Refactoring 계획」을 수립했다.

웹 인터페이스 점검 WebGL 3D 시뮬레이션 사용성 리뷰 2차 Refactoring 계획 시제품 제작
  • 웹 인터페이스 점검 — 컴퓨터실에서 팀원들이 함께 ARES 웹 페이지를 띄워 화면 구성과 동작을 점검.
  • 3D 시뮬레이션 점검 — 홈 화면의 WebGL 3D 연출(탐사로봇 ‘알비’의 우주 유영, 제트팩 우주인)을 확인.
  • 2차 Refactoring 계획 수립 — 점검 결과를 바탕으로 시뮬레이터·피코+블록코딩·웹 인터페이스 세 트랙의 「2차 Refactoring 계획」을 수립. (1차 동작 보존 분할의 연장선)
  • 시제품 제작 — Bambu Lab 3D 프린터로 디자인 부품의 시제품을 출력.

🧱 주요 업무 — 조립형 모델 시작품 제작

최종 제품의 배선 조립과 외형 조립이 쉬워지도록, 본체·외형 패널·상단 발사대 모듈로 분해·결합되는 조립형 모델 시작품을 3D 프린팅으로 제작했다. 내부 구동부와 배선을 먼저 배치한 뒤 측면 외형 패널과 상단 모듈을 끼워 맞추는 모듈형 결합 구조로, 양산 시 조립·점검·유지보수가 용이한지를 실물로 검증했다.

조립형 모델 모듈형 패널 결합 내부 배선 배치 외형 조립 조립성 검증
  • 내부 배선·구동부 배치 — 발사대 하우징 내부에 DC 기어모터·구동축과 배선을 먼저 배치하고, 배선 경로가 외부로 드러나지 않도록 내부 공간을 설계.
  • 모듈형 외형 결합 — 측면 외형 패널(도어)과 상단 발사대 모듈을 분리형으로 만들어, 배선 후 외형을 끼워 맞추는 조립·점검이 쉬운 구조로 검증.
  • 조립성 검증 — 분해→배선→외형 결합→완성의 전 과정을 실물 시작품으로 재현해 양산 조립 흐름을 점검.

🧩 주요 업무 — 시뮬레이션 모듈 분할 리팩터링 (유형 A 실행)

「2차 Refactoring 계획」의 첫 실행으로, 가장 비대했던 simulation.js(2,407줄)의 거대 클로저 buildSim()를 동작 보존 원칙 아래 공유 컨텍스트(ctx) + 서브시스템 모듈 구조로 분해했다. 그 결과 simulation.js는 372줄의 얇은 오케스트레이터로 줄고, 기능은 Web/Sim_Parts/ 13개 모듈로 분리됐다.

유형 A · 거대 클로저 공유 컨텍스트(ctx) Sim_Parts 13개 모듈 2,407 → 372줄 동작 보존
  • 거대 클로저 분해 — buildSim()에 묶여 있던 ~50개 상태와 서브시스템을 Sim_Parts/context.js(씬·카메라·렌더러·ctx) 중심으로 끌어올리고, 각 기능을 모듈로 분리: assets·leds·movement·gun·rocket·traffic·waves·oled·render·audio·dispatch·topics.
  • 얇은 오케스트레이터화 — simulation.js는 setupSimulation()으로 ctx를 만들어 각 모듈에 넘기는 조립 역할만 남김(2,407→372줄). 외부 인터페이스는 그대로라 main.js는 주석 1줄 외 변경 없음.
  • 동작 수정 병행 — 음파 생성(SpawnSoundWave) 보정과 중복 코드 정리 등 소규모 버그 수정 2건 포함.

🤖 주요 업무 — 피코 펌웨어 설정 개편 & 속도 기능

펌웨어의 설정 체계를 교사·어린이가 메모장으로 직접 고치는 텍스트 설정으로 바꾸고, 이동 블록에 속도 조절을 추가했다. 기존 system.json(JSON)을 주석·섹션이 달린 ModelFactorySetting.txt로 대체해, 코드를 몰라도 속도·핀·모듈·탭 이름을 직접 설정할 수 있게 했다.

JSON → 텍스트 설정 ModelFactorySetting.txt 속도 기능 탭 한글 이름 테마 모델(rover/launchpad) 주석 보존 저장
  • 설정 파일 텍스트화 — system_config.py에 텍스트 파서를 추가해 키=값(또는 키:값) 형식을 읽고, 첫 부팅 시 설명이 담긴 기본 템플릿을 자동 생성. 저장 시 기존 주석·가이드라인을 그대로 보존하고 값만 갱신.
  • 속도 기능 — 서보·DC 모터 이동 명령이 속도 인자를 받도록 형식을 확장(SERVO_방향,속도·DC_t방향,초,속도 등). max_speed 기준으로 스케일링하고 0~100%로 제한.
  • 커스터마이징 확장 — 블록코딩 카테고리 탭 한글 이름(GET_NAMES), 테마 모델(로버/발사대), 모듈 활성화 토글, GP 핀 번호, 서보 좌우 보정치를 모두 설정 파일에서 제어.
  • 웹 연동 — 변경된 명령 프로토콜에 맞춰 blocklyconfig.js·bluetooth.js·commandexecutor.js 등 웹 측도 함께 갱신(펌웨어·웹 짝 반영 필요).
