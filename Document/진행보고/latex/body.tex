\section{1일차 — 컴퓨터실 개발 착수}
\textit{2026.06.23 (월) · 10:00\,--\,17:00 KST}

\begin{summarybox}
컴퓨터실에서 개발에 착수했다. 로버 하드웨어(섀시$\cdot$모터 컨트롤러 보드)를
조립하고, 발사대$\cdot$로켓$\cdot$LED 모형을 제작했으며, 블록 코딩과 펌웨어 작업을
병행했다. 동시에 저장소 전체를 코드 수준에서 분석한 종합 보고서를 작성했다.
\end{summarybox}

\subsection{주요 업무 — 코드 분석 종합 보고서 작성}
ARES 저장소~\cite{ares-repo} 전체를 코드 수준에서 분석한 \textbf{「ARES 프로젝트
코드 분석 종합 보고서」(전 14장)}를 작성했다. 웹$\cdot$펌웨어$\cdot$AI 세 런타임
도메인이 \emph{개행 구분 텍스트 명령}이라는 단일 계약으로 느슨하게 결합되는
구조를 중심으로, 하드웨어부터 향후 연구 방향까지 망라했다. 핵심 내용은 본문
「산출물」에 요약하고, 전문은 \ref{app:code}에 수록한다.

\smallskip
\noindent{\raggedright\chip{개요·구성요소} \chip{하드웨어·핀 배치} \chip{피코 펌웨어} %
\chip{웹 서비스 계층} \chip{Web Bluetooth} \chip{블록 코딩} \chip{WebGL 3D 시뮬레이션}\par}

\begin{itemize}
  \item \textbf{대상 시스템} — 초등학생용 블록 코딩 화성 탐사 로버 ‘알비’:
  Blockly 웹앱~\cite{blockly} + 라즈베리파이 피코(MicroPython) 펌웨어 + AI 보조 모듈.
  \item \textbf{핵심 발견} — 웹과 펌웨어는 코드를 공유하지 않고 \emph{텍스트 명령
  프로토콜}만으로 통신해 독립 개발$\cdot$교체가 쉬운 반면, 프로토콜 변경 시 양측
  동시 반영 의존이 생긴다.
  \item \textbf{범위} — 블록 1개가 ‘블록 → 명령 문자열 → BLE → 펌웨어 핸들러 →
  하드웨어’로 도달하는 전 경로를 코드로 추적.
\end{itemize}

\section{2일차 — 교구 제작 협력 업체(코리아사이언스) 방문 \& 웹 서비스 시연}
\textit{2026.06.24 (화) · 10:00\,--\,17:00 KST}

\begin{summarybox}
교구 제작 협력 업체인 코리아사이언스의 생산 공장을 방문해 키트 부품 생산 공정을
견학하고 재고$\cdot$물류를 협의했다. 이어 회의실에서 ARES 웹 서비스를 시연하고
부품$\cdot$앱의 실제 사용 흐름을 함께 검토했다.
\end{summarybox}

\subsection{주요 업무 — 생산 공정 점검 및 웹 서비스 시연}
키트 양산을 담당하는 교구 제작 협력 업체 코리아사이언스의 생산 공장을 방문해 부품 생산
공정을 현장에서 점검하고, 완성된 ARES 블록 코딩 웹 서비스를 관계자에게
시연$\cdot$리뷰했다.

\begin{itemize}
  \item \textbf{생산 공정 견학} — GPL.TECH 레이저 커팅기로 가공되는 목재 키트 부품의
  절단$\cdot$정밀도$\cdot$후처리 공정을 확인.
  \item \textbf{부품$\cdot$재고 점검} — 창고의 키트 부품 재고$\cdot$포장 상태를 보고
  양산$\cdot$물류 일정을 협의.
  \item \textbf{웹 서비스 시연} — 협력업체 관계자에게 ARES 웹 서비스(“블록 코딩으로
  화성을 탐사하다”)를 시연하고 피드백을 수렴, 레이저 커팅 부품과 태블릿 앱의 사용
  흐름을 함께 점검.
\end{itemize}

\section{3일차 — 개발 지속 · 디자인 프로토타입 제작}
\textit{2026.06.25 (수) · 10:00\,--\,17:00 KST}

\begin{summarybox}
컴퓨터실로 복귀해 개발을 이어갔다. 통합$\cdot$검증 작업과 함께 1차 Refactoring
계획을 수립하고, 자체 설계 디자인을 3D 프린터로 출력해 프로토타입을 점검하며

\end{summarybox}

\subsection{주요 업무 — 1차 Refactoring 계획 수립}
지나치게 큰 함수$\cdot$파일로 분석이 어려운 소스를 기능 단위로 분할하기 위한
\textbf{「1차 Refactoring 계획」}을 수립했다. \emph{동작 보존(behavior-preserving)}을
원칙으로, 비대 지점을 네 가지 문제 유형으로 분류하고 6단계 로드맵을
도출했다~\cite{fowler2018refactoring, martin2008cleancode}. 전문은 \ref{app:refactor1}.

  ES module$\cdot$\code{fetch()} 차단, (2) 캐시버스팅 없는 \code{main.js}의 브라우저 캐시}
  가능성으로 좁힘. 재발 방지로 모듈 로드에 버전 쿼리(\code{main.js?v=…}) 적용을 제안.

학습 관제실(개요)부터 미션 화면까지의 화면 동선을 점검해, 어린이가 더 단순하고
직관적으로 쓸 수 있도록 \textbf{인터페이스 개선계획}을 수립했다. 핵심은 세 가지 ---
\textbf{목차 간략화}, \textbf{미션 설명의 제목 노출 + 상세 토글화}, \textbf{중복 메뉴 제거}로,
화면을 덜 복잡하게 하고 코딩 영역을 더 넓게 확보하는 방향이다.

\begin{itemize}
  \item \textbf{목차 간략화} — 개요(학습 관제실)의 ‘차시 한눈에 보기’ 표$\cdot$차시 안내를
  핵심 정보 위주로 압축해, 한눈에 들어오고 차시 선택까지의 동선이 짧아지도록 정리.
  \item \textbf{미션 설명 개선} — 미션 화면에서 \emph{미션 제목은 항상 노출}하고, 상세
  설명(스토리$\cdot$학습 목표)은 \emph{펼침/접힘 토글}로 나타났다 사라지게 한다. 처음엔
  제목만 보여 화면을 단순하게, 필요할 때만 상세를 펼쳐 코딩 영역을 넓게 쓴다.
  \item \textbf{중복 메뉴 삭제} — 같은 기능을 가리키는 중복 버튼(예: 상단바 ‘블록코딩’과
  네비게이터 ‘블록 코딩’ 전환 버튼)을 하나로 정리해 메뉴 혼선을 없앤다.
\end{itemize}

\subsection{주요 업무 — 피코(Pico) 펌웨어 개선 (모터 속도 제한 · 미션 맥락 블록 제한)}
펌웨어(라즈베리파이 피코) 측 개선으로, 안전하고 다루기 쉬운 주행을 위해 \textbf{모터
속도 제한 코드를 추가}했다(개선 결과). 나아가 어린이가 미션에 집중하도록 \textbf{미션
맥락에 따라 노출 블록을 제한}하는 개념을 실현하는 것을 다음 개선 방향으로 설정했다.

\begin{itemize}
  \item \textbf{[개선 결과] 모터 속도 제한 코드 추가} — DC$\cdot$서보 모터 이동 명령이 받는
  속도를 \code{max\_speed} 기준으로 스케일링하고 \textbf{0\,--\,100\% 범위로 제한}해, 과속으로
  인한 이탈$\cdot$안전 문제를 막고 어린이가 다루기 쉬운 일정한 주행을 보장.
  \item \textbf{[개선 방향] 미션 맥락에 따른 노출 블록 제한} — 각 차시$\cdot$미션의 학습
  목표에 필요한 블록만 블록코딩 툴박스에 노출하고 나머지는 숨기는 개념을 실현. 어린이가
  해당 미션에 집중하도록 돕는 \emph{학습 스캐폴딩}으로, 펌웨어의 미션/모델 설정과 웹
  툴박스를 연계해 단계적으로 적용할 계획.
\end{itemize}

\subsection{주요 업무 — 시뮬레이션 개선 (객체 배치 완료 · property 기반 명령 연동)}
3D 시뮬레이션에서 장면에 \textbf{객체(로봇$\cdot$장치$\cdot$소품 등)를 배치하는 작업을 완성}했다(개선
결과). 다음 단계로, 각 객체에 \textbf{property로 ‘연동 명령’을 지정}해 두고 \textbf{해당
명령이 들어오면 그 명령에 맞는 동작을 수행}하도록 하는 데이터 주도 방식을 개선
방향으로 잡았다.

\begin{itemize}
  \item \textbf{[개선 결과] 시뮬레이션 객체 배치 완성} — 시뮬레이션 장면에 필요한
  객체(탐사로봇$\cdot$장치$\cdot$소품 등)를 원하는 위치에 배치하는 작업을 완료.
  \item \textbf{[개선 방향] property 기반 명령 연동} — 각 객체에 \emph{property로 연동
  명령을 지정}할 수 있게 하여, 그 명령이 수신되면 객체가 그에 맞는 동작(점등$\cdot$회전$\cdot$%
  이동$\cdot$발사 등)을 수행하도록 한다. 객체마다 코드를 따로 짜지 않고 \textbf{property 설정만으로
  ‘명령 $\to$ 동작’을 매핑}하는 선언적 구조라, 새 객체$\cdot$미션을 추가하기 쉽고 블록 명령과
  시뮬레이션 동작의 일관성도 높아진다.
\end{itemize}

\subsection{주요 업무 — 개선계획 토론 결과 (트랙별 다음 개선 방향)}
팀 토론을 통해 세 트랙(\textbf{웹 $\cdot$ 피코 $\cdot$ 시뮬레이션})의 다음 개선 방향을
구체화했다. 어린이가 더 빠르게 블록코딩에 몰입하고, 블록이 직관적이며, 시뮬레이션
객체를 효율적으로 다루도록 하는 데 초점을 맞췄다.

\begin{itemize}
  \item \textbf{[웹] 사이드 인터페이스 구현} — 차시(목차)를 길게 노출하기보다,
  \textbf{사이드 형태의 인터페이스}로 \textbf{블록코딩 화면에 빠르게 유도$\cdot$집중}시키는
  인터페이스 개발이 필요(앞선 ‘목차 간략화$\cdot$중복 메뉴 제거’의 연장선).
  \item \textbf{[피코] 펌웨어 거의 완성 --- 블록코딩 보완 필요} — 피코 펌웨어는 거의
  완성 단계. 다만 블록코딩에서 \textbf{수학/논리 탭의 블록들이 직관적이지 않아} 개선이
  필요하고, \textbf{제어 관련 블록들}도 더 다듬어야 한다.
  \item \textbf{[시뮬레이션] 메시 배치 기능 개선} — 시뮬레이션 \textbf{메시 배치 기능}을
  개선하되, \textbf{메시 hierarchy(계층 구조)를 도입}해 객체를 효율적으로 모델링할 수
  있는 도구를 완성(객체별 ‘property 기반 명령 연동’ 방향과 함께 추진).
\end{itemize}

\subsection{수정 계획 — 치명(HIGH) 4건}
발견 당일 4건의 수정 계획을 수립했다. 원칙은 \textbf{(a) 근본 원인 제거, (b) 웹$\cdot$펌웨어 양쪽
방어, (c) 자동화 검증 동반}이다.

\begin{itemize}
  \item \textbf{① 자석 반전} — 대시보드 판정 조건을 펌웨어 규약(1=감지)에 맞게 뒤집는다(한 줄).

  \item \textbf{④ 비상정지 즉시화} — 웹: 비상정지 시 \textbf{대기 중 응답 promise를 즉시 취소}해

  바꾸고 매 슬라이스 UART를 폴링해 STOP\_ALL 도착 시 즉시 정지. \textbf{시간 상한 60초}를

\end{itemize}

    """50ms 슬라이스로 대기하며 매 슬라이스 비상정지 도착을 확인한다."""
    if sec > MAX_TIMED_SEC:

        utime.sleep_ms(50)
    return True

\end{itemize}

\subsection{서비스 앱 --- 맥락 인지형 중앙 버튼 UI}
디자인 점검에서 이어, 서비스 앱 하단 \textbf{중앙 버튼}을 \textbf{맥락 인지형(context-aware)}으로
구현했다. 같은 자리의 버튼 하나가 \textbf{사용자의 상황(화면 모드$\cdot$연결 상태$\cdot$실행 여부)에
따라 색$\cdot$아이콘$\cdot$라벨$\cdot$동작을 스스로 바꿔} 그때 필요한 조작만 노출한다 --- 연결이 안 됐으면
실행 대신 연결을, 시뮬레이션은 연결 없이 모의실행을 안내하는 식이다. 아래 상태 표는 실제 화면
캡처이고, 오른쪽은 실기(모니터+발사대) \textbf{동영상에서 발췌}했다.

\smallskip
\noindent{\raggedright\chip{상황별 자동 전환} \chip{신호연결↔연결됨} \chip{실행↔비상정지} %
\chip{모의실행↔실험중단} \chip{연결 없이 시뮬}\par}

\begin{itemize}
  \item \textbf{미연결} --- 빨강 `\textbf{신호연결}'(링크 아이콘). 누르면 탐사선 연결을 시도.
  \item \textbf{연결됨}(비코딩) --- 초록 `\textbf{연결됨}'.
  \item \textbf{코딩 모드$\cdot$연결} --- 골드(\code{\#E6AD00}) `\textbf{실행}'(재생 아이콘). 블록을 로버로
  전송. 코딩 모드라도 \textbf{미연결이면 `신호연결'}로 표시(실행 불가).
  \item \textbf{코딩 실행 중} --- \code{\#B94A45} `\textbf{비상정지}'(정지 아이콘). 누르면 즉시 정지(STOP\_ALL).
  \item \textbf{시뮬레이션 모드} --- 연결 없이 블루(\code{\#3F78A8}) `\textbf{모의실행}'(재생 아이콘).
  \item \textbf{시뮬 실행 중} --- \code{\#B94A45} `\textbf{실험중단}'(정지 아이콘). 누르면 시뮬 중지.
\end{itemize}

\subsection{서비스 앱 --- 동작 품질 테스트 (QA 체계$\cdot$통합 테스트 명세서)}
컴퓨터실에서 \textbf{서비스 앱 동작 품질 테스트}를 체계화했다. \textbf{V-모델 기반 QA 체계}
(요구사항 분석 $\to$ 시스템 설계 $\to$ 상세 설계 $\to$ 구현 $\to$ 단위/통합/시스템/인수 테스트)와
역할(R\&R)을 정리한 \textbf{품질 보증 명세서}를 작성하고, 이를 근거로 \textbf{통합 테스트 케이스
명세서 v1.0}(\textbf{14개 카테고리 $\cdot$ 98개 실행형 케이스})을 도출했다. 케이스는 \textbf{Pico
펌웨어 10개 모듈 $\cdot$ Web 블록코딩 실행엔진 $\cdot$ 3D 시뮬레이터 $\cdot$ XML Presets}를 전수
분석해, 각 케이스를 실제 코드의 명령$\cdot$블록 흐름에 대응시켜 \textbf{Pass/Fail 기준}과 함께
서술했다. 시뮬레이터로는 실물 기기 없이도 케이스를 실행$\cdot$검증할 수 있게 했다.
