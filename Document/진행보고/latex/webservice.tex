\chapter{프로젝트 개시 시점 개발 준비 상태}
6월 현장 착수 이전, \textbf{2025년 겨울부터 준비단계}에 구현해 GitHub Pages로 배포한
ARES 웹 서비스(\href{https://dknife.github.io/ARES2}{\code{dknife.github.io/ARES2}})의
주요 기능을 정리한다. \textbf{이후 1$\sim$11일차의 현장 개발은 이 준비단계 결과물을
기반으로} 진행됐다. 설치 없이 브라우저만으로 학습 콘텐츠 열람, 블록 코딩, 3D
시뮬레이션, AI 도움, 기기 모니터링이 가능하며, 블록 1개가 명령 문자열로 변환되어
BLE\footnote{\textbf{BLE(Bluetooth Low Energy)}: 저전력 블루투스. 적은 전력으로 소량의
데이터를 근거리 무선으로 주고받는 규격으로, 웹에서는 Web Bluetooth API로 접근한다.}로
로버에 전달되는 전체 경로(\ref{app:code})를 실물 기기 없이도 시뮬레이션으로
체험할 수 있다.

\section{블록 코딩과 3D 시뮬레이션}
미션을 고르면 Google Blockly\footnote{\textbf{Blockly}: Google이 개발한 오픈소스 시각적
프로그래밍 라이브러리. 퍼즐 형태의 블록을 끼워 맞춰 프로그램을 구성하며, 조립된 블록은
실행 가능한 코드나 명령 문자열로 변환된다.} 기반 블록 코딩 화면이 열린다. 좌측 툴박스는
서보$\cdot$DC 모터$\cdot$LED$\cdot$디스플레이$\cdot$소리$\cdot$발사$\cdot$센서$\cdot$시간$\cdot$제어$\cdot$변수$\cdot$수학$\cdot$함수
등 도메인별 카테고리로 구성된다. ‘예제’를 불러오면(아래는 \emph{장애물 피하기})

\section{관련 연구 및 배경}

2025년 겨울 준비단계에서 ARES 팀은 프로젝트 개시 시점에 이미 상당 수준의 웹 서비스를 GitHub Pages로 배포해 둔 상태였다. 이 서비스의 가장 두드러진 특징은 \emph{설치가 필요 없는} 교육 플랫폼이라는 점이다. 학습자는 브라우저 주소창에 URL을 입력하는 것만으로 Blockly 블록코딩, Web Bluetooth 기반 로버 제어, WebGL 3D 시뮬레이션, AI 도움말, 점검 대시보드에 즉시 접근할 수 있다. 이러한 무설치 배포 모델은 초등학교 컴퓨터실처럼 관리자 권한이 제한되고 기기 사양이 균일하지 않은 환경에서 결정적 이점을 지닌다. 웹을 배포 매체로 삼는 결정은 접근성과 도달 범위를 극대화하려는 설계 판단이며, 사용성 공학이 강조해 온 낮은 초기 마찰의 원칙과 부합한다~\cite{nielsen1993usability, krug2014dontmakemethink}.

블록코딩 계층은 Google Blockly를 채택했다. Blockly는 시각적 블록을 조립해 프로그램을 구성하는 오픈소스 라이브러리로, Scratch가 대중화한 블록 기반 프로그래밍의 계보를 잇는다~\cite{blockly, maloney2010scratch}. 준비단계 서비스에서 이 계층은 이미 어린이가 드래그로 명령을 배열하고 이를 실행 가능한 형태로 번역할 수 있도록 구성되어 있었다. 블록코딩은 문법 오류라는 초심자의 대표적 좌절 요인을 구조적으로 제거함으로써, 학습자가 알고리즘적 사고 자체에 집중하도록 돕는다. 이는 계산적 사고 교육 연구가 반복해 확인한, 구문 부담의 제거가 개념 학습을 촉진한다는 관찰과 일치한다~\cite{brennan2012ct, grover2013ct}.

실물 제어 계층은 Web Bluetooth API를 통해 라즈베리파이 피코 로버와 통신한다. 브라우저가 직접 BLE 장치에 연결하여 명령을 주고받는 이 방식은, 별도의 네이티브 앱이나 드라이버 설치 없이 하드웨어 상호작용을 가능하게 한다는 점에서 무설치 철학을 하드웨어 영역까지 확장한다~\cite{webbluetooth, bluetooth54}. 다만 Web Bluetooth는 보안상 \code{localhost} 또는 HTTPS 문맥에서만 동작하므로, GitHub Pages의 HTTPS 배포는 단순한 호스팅 선택을 넘어 이 기능을 성립시키는 전제 조건이었다. 준비단계에서 이 제약을 만족하는 배포 파이프라인을 이미 갖춘 점은, 개시 시점의 기술적 성숙도를 보여 주는 지표다.

시각화 계층은 Three.js 기반 WebGL 3D 시뮬레이터로 구현되었으며, 준비단계에 이미 네 종의 시뮬레이션 모델을 제공했다. 시뮬레이터는 실물 로버가 없는 학습자도 동일한 블록 프로그램을 가상 환경에서 실행해 볼 수 있게 하여, 하드웨어 보급의 병목을 우회한다. 실시간 렌더링을 브라우저 안에서 수행하는 이 구조는 WebGL과 glTF 자산 파이프라인, 그리고 실시간 렌더링의 표준적 기법 위에 서 있다~\cite{threejs, webgl, gltf, akenine2018realtime}. 물리 로버와 가상 시뮬레이터가 같은 명령 프로토콜을 공유하도록 설계된 덕분에, 학습자는 매끄럽게 두 대상 사이를 오갈 수 있다.

끝으로 준비단계 서비스는 로컬 LLM 기반 AI 도움말과 시스템 점검 대시보드를 포함해 학습 지원과 운영 관리를 아울렀다. AI 도우미는 어린이가 블록코딩 인터페이스에 관해 던지는 질문에 답하도록 설계되어, 교사의 개입 없이도 즉각적 안내를 제공한다~\cite{exaone35, transformers}. 대시보드는 로버의 연결 상태와 모듈 활성 여부를 한눈에 보여 주어 현장 운영의 신뢰성을 뒷받침한다. 이처럼 학습, 제어, 시각화, 지원, 점검이라는 다섯 축을 하나의 무설치 웹 서비스로 통합한 상태가 곧 프로젝트 개시 시점의 출발선이었다.

\section{구현을 위한 의사코드}

준비단계 서비스의 핵심 운영 논리는, 접속한 사용자의 환경을 감지하여 실물 로버 연결과 가상 시뮬레이터 사이에서 자연스럽게 학습 경로를 안내하는 진입 흐름이다. 아래 의사코드는 브라우저가 Web Bluetooth를 지원하고 안전 문맥(HTTPS)에서 실행되는지를 먼저 판정한 뒤, 실물 연결을 시도하되 실패하거나 장치가 없을 때는 시뮬레이터로 우아하게 대체하는 절차를 기술한다. 이 흐름이 무설치 접근성과 하드웨어 보급 병목 우회라는 두 목표를 동시에 만족시킨다.

\captionof{listing}{의사코드 --- 준비단계 서비스 진입 흐름}
\begin{minted}{text}
함수 학습세션_시작(브라우저환경):
    만약 브라우저환경이 안전문맥이_아니면:     # http, file:// 등
        메시지_표시("HTTPS 주소로 접속해 주세요")
        시뮬레이터_모드로_전환()
        반환 "시뮬레이션"
    만약 웹블루투스_지원(브라우저환경)이면:
        장치 = 로버_연결_시도()
        만약 장치가 연결됨이면:
            반환 "실물제어"
        아니면:
            시뮬레이터_모드로_전환()      # 장치 없거나 취소
            반환 "시뮬레이션"
    아니면:
        시뮬레이터_모드로_전환()
        반환 "시뮬레이션"
\end{minted}

이 진입 흐름의 요점은 어떤 경로로 진입하든 학습자가 \emph{동일한 블록 편집 경험}에 도달한다는 데 있다. 실물 제어와 시뮬레이션은 모두 같은 개행 구분 텍스트 명령을 소비하므로, 진입 판정은 명령의 목적지만 바꿀 뿐 학습 인터페이스 자체는 변하지 않는다. 덕분에 로버가 있는 교실과 없는 교실이 단일한 커리큘럼을 공유할 수 있으며, 이는 준비단계 서비스가 지향한 보편적 접근성의 구현적 표현이다.

\section{기술적·학술적 의의}

기술적으로 준비단계 서비스는 무설치 웹 배포가 어디까지 교육 플랫폼의 기능을 담아낼 수 있는지를 보여 준 실증 사례다. 통상 하드웨어 제어와 3D 시뮬레이션은 네이티브 애플리케이션의 영역으로 여겨지지만, ARES는 Web Bluetooth와 WebGL을 결합해 이를 순수 브라우저 문맥으로 끌어들였다~\cite{webbluetooth, webgl}. GitHub Pages라는 정적 호스팅 위에서 HTTPS 배포와 다섯 계층 통합을 달성한 점은, 최소한의 운영 인프라로 최대한의 기능을 전달하려는 실용적 설계의 성과다.

교육적 의의는 접근 장벽의 근본적 제거에 있다. 설치, 계정, 관리자 권한이 필요 없는 서비스는 컴퓨터실 환경의 이질성과 관리 제약을 뛰어넘어 모든 학습자에게 균일한 출발점을 제공한다. 특히 네 종의 시뮬레이션 모델과 실물 로버가 같은 프로토콜을 공유하도록 설계함으로써, 하드웨어 보유 여부와 무관하게 동일한 계산적 사고 활동을 지원한다. 이는 낮은 진입 장벽과 높은 천장을 동시에 추구하는 구성주의 학습 환경의 이상에 부합한다~\cite{papert1980, resnick2009scratch}.

학술적으로 이 장은 프로젝트 개시 시점의 기술 자산을 명시적으로 기록한다는 점에서 가치가 있다. 개발은 백지에서 시작하지 않았고, 이미 배포되어 동작하는 다중 계층 웹 서비스라는 기반 위에서 진행되었다. 이 출발선을 정확히 서술해 두어야 이후의 개선과 확장이 어떤 토대에서 이루어졌는지 평가할 수 있으며, 이는 소프트웨어 프로젝트의 진화를 서술하는 데 필수적인 기준점이 된다~\cite{mcconnell2004codecomplete}. 따라서 준비단계 상태의 기록은 단순한 현황 보고를 넘어, 프로젝트 전체 서사의 좌표계를 설정하는 학술적 기능을 수행한다.

\section{설계 대안·정량 분석·확장 논의}

ARES 개발 개시 시점의 가장 근본적인 결정은 \emph{배포 매체}의 선택이었다. 초등학생 대상 교구는 학교 관리 PC, 태블릿, 가정용 노트북 등 통제 불가능한 이기종 환경에서 실행되므로, 설치·업데이트·권한 문제가 교사의 도입 장벽을 좌우한다. 우리는 세 가지 대안을 저울질했다. 첫째, 운영체제별 \textbf{네이티브 앱}은 GPU와 블루투스 스택에 대한 최상의 성능·접근성을 주지만 iOS/Android/Windows/ChromeOS 각각의 스토어 심사와 서명, 버전 파편화를 감당해야 한다. 둘째, \textbf{순수 웹}은 브라우저만 있으면 무설치로 즉시 실행되고 URL 하나로 배포·갱신이 끝나지만, Web Bluetooth\cite{webbluetooth}가 Chromium 계열에만 구현되어 있고 보안 컨텍스트(HTTPS/localhost)를 요구한다는 제약이 있다. 셋째, \textbf{하이브리드}(WebView 래핑)는 무설치성을 일부 포기하는 대신 네이티브 BLE 브리지를 확보한다. 우리는 \emph{교사가 실물 로버 없이도 시뮬레이터만으로 수업 대부분을 진행할 수 있다}는 점을 결정적 근거로 삼아 순수 웹을 택했다. 즉 BLE가 없는 환경에서도 Blockly\cite{blockly} 블록 코딩과 Three.js\cite{threejs} 3D 시뮬레이션이 온전히 동작하고, 실물 연결은 지원 브라우저에서만 추가로 활성화되는 \emph{점진적 향상(progressive enhancement)} 구조다\cite{marcotte2010responsive}.

이 선택의 타당성은 정량적으로도 뒷받침된다. 무설치 웹은 배포 지연이 사실상 0에 수렴한다. 코드 수정 후 재배포까지 스토어 심사(수 시간\char`~수 일)가 개입하는 네이티브와 달리, 정적 자산을 갱신하고 캐시를 무효화하는 것으로 전 사용자에게 즉시 반영된다. 접근성 측면에서는 WCAG 2.1\cite{wcag21}의 명도 대비 4.5:1 기준과 키보드 조작 가능성을 브라우저 표준 위에서 그대로 상속받을 수 있어, 자체 UI 프레임워크를 만들 때보다 준수 비용이 낮다. 성능 측면에서 관건은 3D 시뮬레이터의 초기 로딩과 프레임률이었다. 4종 시뮬 모델(알비/신호등/발사대/로버)을 glTF\cite{gltf}로 배포하되 meshopt\cite{meshopt} 압축과 텍스처 다운스케일(2048$\to$1024)을 적용해 전송량을 줄였고, 실시간 렌더링 기법\cite{akenine2018realtime}을 참고해 저사양 내장 GPU에서도 상호작용 프레임률을 확보하는 것을 목표로 했다. 이때 \emph{자산 용량}과 \emph{시각적 충실도}는 정면으로 충돌하는 축이며, 우리는 초등 교육이라는 맥락에서 후자보다 전자(빠른 첫 화면)를 우선했다.

\captionof{listing}{의사코드 --- 실행 환경 능력 탐지와 점진적 향상}
\begin{minted}{text}

    만약 보안_컨텍스트가_아니면:

한계와 확장 방향은 명확하다. 현재 구조의 가장 큰 취약점은 \textbf{네트워크 의존성}과 \textbf{브라우저 종속성}이다. 학교 무선망이 불안정하거나 방화벽이 CDN을 차단하면 첫 로딩이 실패할 수 있으므로, 향후 \emph{오프라인 우선(offline-first)} 확장이 필요하다. 구체적으로는 Service Worker와 캐시 스토리지를 이용해 최초 접속 시 블록 에디터·시뮬 자산·AI 도우미 리소스를 통째로 캐싱하고, 이후에는 완전 오프라인으로 실행되는 설치형 웹앱(PWA) 형태가 유력하다. 실제로 프로젝트는 이미 \code{file://} 오프라인 빌드 경로를 유지해 씬·자산을 인라인으로 묶는 방식을 병행하고 있어\cite{ares-repo}, 이를 표준 PWA 규격으로 승격하는 것이 자연스러운 다음 단계다. 둘째, Web Bluetooth 미지원 브라우저(Safari/Firefox) 사용자를 위해 하이브리드 래퍼를 \emph{선택적} 배포물로 추가하되, 코드베이스는 단일 웹 소스를 공유하도록 유지하는 이원 전략이 비용 대비 효과가 크다.

마지막으로, \emph{물리 로버와 가상 시뮬레이터가 동일한 개행 구분 텍스트 명령 프로토콜을 공유}한다는 초기 설계 결정은 이후 모든 장(章)에 파급되는 지렛대였다. 명령 문자열 하나가 실물 UART(9600 baud)로 나가든 시뮬레이터의 \code{CommandDispatcher}로 들어가든 동일하게 해석되므로, 아동은 시뮬레이션에서 익힌 블록을 그대로 실물에 적용할 수 있고 개발자는 프로토콜을 한 곳에서만 관리한다. 이는 소프트웨어 공학에서 말하는 \emph{인터페이스를 통한 결합도 최소화}\cite{martin2008cleancode}의 교육용 구현이며, 시뮬레이터를 실물의 \emph{계약 테스트(contract test)} 도구로 재사용할 여지까지 연다. 다만 이 대칭성은 실물 하드웨어의 타이밍·오차(모터 관성, 센서 노이즈)를 시뮬레이터가 완벽히 재현하지 못한다는 근본적 한계를 내포하며, 향후 실측 데이터를 반영한 물리 모델 보정이 충실도 격차를 좁히는 확장 과제로 남는다.
