1차 Refactoring 계획

지나치게 큰 함수·파일로 구성된 소스를 기능 분석을 통해 체계적으로 분할하기 위한 설계 문서입니다. 동작 보존(behavior-preserving) 리팩터링을 원칙으로 하며, 기능 변경·버그 수정은 포함하지 않습니다.

02 — Patterns

문제 유형과 대응 패턴

비대 지점은 네 가지 형태로 나뉘며, 각 형태마다 다른 분할 전략을 적용합니다.

유형 A

거대 클로저에 갇힌 서브시스템

simulation.js — 상태 ~50개가 클로저에 묶여 외부에 안 보임. 함수만 떼면 상태 공유가 깨짐.

대응: 공유 컨텍스트(ctx) + 서브시스템 모듈화. ctx 객체 + 순수 함수 모듈로 14개 분할.

유형 B

거대 디스패처 + 핸들러 더미

process_data.py, generateCommand() — 하나의 분기문이 50개 명령을 처리.

대응: 도메인별 핸들러 모듈 + prefix→handler dict 디스패치. 170줄 → ~30줄.

유형 C

데이터 덩어리 / 만능 함수

blocklyconfig.js, matchAction() — 이미 주석으로 카테고리가 갈려 있음.

대응: 카테고리별 물리 파일 분리 + index.js 로 재조립.

유형 D

인라인 자산

dashboard.html, main.html — HTML 안에 CSS·JS 인라인.

대응: CSS/JS 외부 파일로 추출. 가장 쉽고 위험 낮음, 즉시 효과.

유형 A 상세설명 — "거대 클로저에 갇힌 서브시스템"의 의미

이 한 줄이 가장 추상적이라 실제 simulation.js 코드로 풀어 설명합니다.

1 — 지금 simulation.js가 생긴 모양

현재 buildSim() 함수 하나 안에 모든 게 들어 있습니다. 단순화하면:

function buildSim(THREE, ...) {
    // ── 상태 변수 ~50개가 전부 이 함수 안에 선언됨 ──
    let scene, camera, renderer, worldGroup;
    let eyeL, eyeR, chestLed, roverLeds = [];     // LED
    let servoOn, servoDir, wheelR, wheelL;        // 이동
    let gunMesh, muzzleFlash, muzzleSparks = [];  // 총
    let rocketGroup, rocketLaunchOn, savedCamPos; // 로켓
    // … 등등 ~50개

    // ── 함수들도 전부 이 안에 중첩 ──
    function makeLed()        { ...eyeL, roverLeds...   }   // 바깥 변수를 직접 사용
    function setGunFire()     { ...gunMesh, muzzleFlash... }
    function setRocketLaunch(){ ...rocketGroup...        }
    function render()         { ...거의 모든 변수를 다 읽음... }

    return { makeLed, setGunFire, setRocketLaunch, ... }; // ~50개 반환
}

안쪽 함수들이 바깥 변수 eyeL·gunMesh를 그냥 이름으로 직접 읽고 씁니다. 이게 "클로저(closure)에 묶였다" — 함수가 자기를 감싼 바깥 변수를 기억하는 자바스크립트 특성 — 의 뜻입니다.

2 — "상태 50개가 클로저에 묶여 외부에 안 보임"

gunMesh·rocketGroup 같은 변수는 buildSim 함수 밖에서는 존재 자체를 모릅니다. 그래서 단순히 gun.js로 함수만 떼어내면:

// gun.js (잘못된 분리)
export function setGunFire() {
    muzzleFlash.visible = true;   // ❌ muzzleFlash 가 누구야? 여기선 안 보임!
}

에러. muzzleFlash는 원래 buildSim 안의 변수라 접근 불가. 이게 "함수만 떼면 상태 공유가 깨진다"는 의미입니다.

3 — 대응책: 공유 컨텍스트(ctx)

흩어진 50개 변수를 객체 하나에 모아서, 그 객체를 각 함수에 인자로 넘깁니다. 그 객체가 ctx(context).

// context.js — 공유 상태를 한 객체에 모음
export function createContext(THREE) {
    return {
        scene: null, camera: null, renderer: null, worldGroup: null,
        led:    { eyeL: null, eyeR: null, roverLeds: [] },
        motion: { servoOn: false, wheelR: null, wheelL: null },
        gun:    { mesh: null, muzzleFlash: null, sparks: [] },
        rocket: { group: null, launchOn: false, savedCamPos: null },
    };
}

// gun.js — ctx 를 인자로 받음 (이제 muzzleFlash 가 보임!)
export function setGunFire(ctx) {
    ctx.gun.muzzleFlash.visible = true;   // ✅ ctx 통해 접근
}

// simulation.js — 얇은 조립자(orchestrator)
const ctx = createContext(THREE);   // 상태 객체 하나 생성
setGunFire(ctx);                    // 모두 같은 ctx 를 공유
setRocketLaunch(ctx);

모든 모듈이 같은 ctx 객체를 돌려가며 읽고 쓰니, 파일이 14개로 쪼개져도 상태가 안 깨집니다.

4 — "순수 함수 모듈"이란

각 모듈 함수가 자기 안에 숨은 전역 상태를 갖지 않고, 필요한 것(ctx, dt)을 전부 인자로 받아서 동작한다는 뜻입니다. 그래서 ① 무슨 데이터를 쓰는지 인자만 봐도 드러나고 ② 따로 떼어 테스트하기 쉽고 ③ 숨은 의존이 없습니다. (Three.js 객체를 변이하므로 완전한 수학적 순수함수는 아니고, "전역 의존 없이 인자 기반으로 동작"하는 느슨한 의미입니다.)

5 — 한눈에 Before / After
[ Before ]                       [ After ]
buildSim() 2333줄                 simulation.js (조립자, ~250줄)
 ├ 상태 50개 (클로저)      →        └ ctx 하나 생성 후 모듈에 넘김
 ├ makeLed()                       sim/context.js  ← 상태 컨테이너
 ├ setGunFire()                    sim/leds.js     ← setEye(ctx,...)
 ├ setRocketLaunch()               sim/gun.js      ← setGunFire(ctx)
 └ … (전부 한 함수 안)             sim/rocket.js, movement.js … (총 14개)
비유: 지금은 50명이 한 방에서 책상 없이 바닥에 물건을 늘어놓고 일하는 상태입니다(누가 뭘 쓰는지 알 수 없고, 방을 나누면 물건이 흩어짐). ctx는 공용 캐비닛 하나를 두고, 작업자(모듈)들이 각자 방으로 흩어지되 필요할 때 같은 캐비닛을 열어 쓰는 방식입니다.

위험도가 "높음"인 이유: 50개 변수를 어느 모듈에 둘지 정확히 분류해 ctx 구조를 설계하고, 함수 시그니처를 전부 (ctx, ...)로 바꿔야 해서 손이 많이 갑니다. 그래서 로드맵 맨 마지막(6단계)에 둡니다.

05 — Cross-cutting Concerns

횡단 주의점

분할이 깨뜨리기 쉬운 결합 지점들입니다.

  • 블록 타입 이름 3중 결합 — blocklyconfig.js의 type ↔ commandexecutor.generateCommand()의 case ↔ ai_helper.KNOWN_TYPES. 한곳(상수)으로 모아 단일 출처(SSOT)화하면 동기화 버그를 줄입니다.
  • state.js 전역 의존 — 거의 모든 Web 모듈이 state를 직접 변이. 분할 시 누가 무엇을 쓰는지 드러나므로 이 기회에 접근 경로를 정리합니다.
  • firmware ↔ web 문자열 프로토콜 — 컴파일 검증이 안 됨. Document/API_DOCUMENTATION.md가 유일한 계약 → 변경 시 동시 갱신.
  • 알려진 버그 동거 — wheel turn 로직, SYS_SET 고아 코드 등 기존 버그는 분할과 분리해서 다룹니다. 리팩터링 커밋에 버그 수정을 섞으면 회귀 추적이 어려워집니다.
  • MicroPython RAM 제약 — Pico에서는 파일 분할 시 import 오버헤드가 늘어납니다. 도메인 5~6개 수준으로만 나누고 과분할은 피합니다.
