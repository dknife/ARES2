1차 Refactoring 계획

지나치게 큰 함수·파일로 구성된 소스를 기능 분석을 통해 체계적으로 분할하기 위한 설계 문서입니다. 동작 보존(behavior-preserving) 리팩터링을 원칙으로 하며, 기능 변경·버그 수정은 포함하지 않습니다.

02 — Patterns

문제 유형과 대응 패턴

비대 지점은 네 가지 형태로 나뉘며, 각 형태마다 다른 분할 전략을 적용합니다.

유형 A

거대 클로저에 갇힌 서브시스템

simulation.js — 상태 ~50개가 클로저에 묶여 외부에 안 보임. 함수만 떼면 상태 공유가 깨짐.

대응: 공유 컨텍스트(ctx) + 서브시스템 모듈화. ctx 객체 + 순수 함수 모듈로 14개 분할.

유형 B

거대 디스패처 + 핸들러 더미

process_data.py, generateCommand() — 하나의 분기문이 50개 명령을 처리.

대응: 도메인별 핸들러 모듈 + prefix→handler dict 디스패치. 170줄 → ~30줄.

유형 C

데이터 덩어리 / 만능 함수

blocklyconfig.js, matchAction() — 이미 주석으로 카테고리가 갈려 있음.

대응: 카테고리별 물리 파일 분리 + index.js 로 재조립.

유형 D

인라인 자산

dashboard.html, main.html — HTML 안에 CSS·JS 인라인.

대응: CSS/JS 외부 파일로 추출. 가장 쉽고 위험 낮음, 즉시 효과.

유형 A 상세설명 — "거대 클로저에 갇힌 서브시스템"의 의미

이 한 줄이 가장 추상적이라 실제 simulation.js 코드로 풀어 설명합니다.

1 — 지금 simulation.js가 생긴 모양

현재 buildSim() 함수 하나 안에 모든 게 들어 있습니다. 단순화하면:

function buildSim(THREE, ...) {
    // ── 상태 변수 ~50개가 전부 이 함수 안에 선언됨 ──
    let scene, camera, renderer, worldGroup;
    let eyeL, eyeR, chestLed, roverLeds = [];     // LED
    let servoOn, servoDir, wheelR, wheelL;        // 이동
    let gunMesh, muzzleFlash, muzzleSparks = [];  // 총
    let rocketGroup, rocketLaunchOn, savedCamPos; // 로켓
    // … 등등 ~50개

    // ── 함수들도 전부 이 안에 중첩 ──
    function makeLed()        { ...eyeL, roverLeds...   }   // 바깥 변수를 직접 사용
    function setGunFire()     { ...gunMesh, muzzleFlash... }
    function setRocketLaunch(){ ...rocketGroup...        }
    function render()         { ...거의 모든 변수를 다 읽음... }

    return { makeLed, setGunFire, setRocketLaunch, ... }; // ~50개 반환
}

안쪽 함수들이 바깥 변수 eyeL·gunMesh를 그냥 이름으로 직접 읽고 씁니다. 이게 "클로저(closure)에 묶였다" — 함수가 자기를 감싼 바깥 변수를 기억하는 자바스크립트 특성 — 의 뜻입니다.

2 — "상태 50개가 클로저에 묶여 외부에 안 보임"

gunMesh·rocketGroup 같은 변수는 buildSim 함수 밖에서는 존재 자체를 모릅니다. 그래서 단순히 gun.js로 함수만 떼어내면:

// gun.js (잘못된 분리)
export function setGunFire() {
    muzzleFlash.visible = true;   // ❌ muzzleFlash 가 누구야? 여기선 안 보임!
}

에러. muzzleFlash는 원래 buildSim 안의 변수라 접근 불가. 이게 "함수만 떼면 상태 공유가 깨진다"는 의미입니다.

3 — 대응책: 공유 컨텍스트(ctx)

흩어진 50개 변수를 객체 하나에 모아서, 그 객체를 각 함수에 인자로 넘깁니다. 그 객체가 ctx(context).

// context.js — 공유 상태를 한 객체에 모음
export function createContext(THREE) {
    return {
        scene: null, camera: null, renderer: null, worldGroup: null,
        led:    { eyeL: null, eyeR: null, roverLeds: [] },
        motion: { servoOn: false, wheelR: null, wheelL: null },
        gun:    { mesh: null, muzzleFlash: null, sparks: [] },
        rocket: { group: null, launchOn: false, savedCamPos: null },
    };
}

// gun.js — ctx 를 인자로 받음 (이제 muzzleFlash 가 보임!)
export function setGunFire(ctx) {
    ctx.gun.muzzleFlash.visible = true;   // ✅ ctx 통해 접근
}

// simulation.js — 얇은 조립자(orchestrator)
const ctx = createContext(THREE);   // 상태 객체 하나 생성
setGunFire(ctx);                    // 모두 같은 ctx 를 공유
setRocketLaunch(ctx);

모든 모듈이 같은 ctx 객체를 돌려가며 읽고 쓰니, 파일이 14개로 쪼개져도 상태가 안 깨집니다.

4 — "순수 함수 모듈"이란

각 모듈 함수가 자기 안에 숨은 전역 상태를 갖지 않고, 필요한 것(ctx, dt)을 전부 인자로 받아서 동작한다는 뜻입니다. 그래서 ① 무슨 데이터를 쓰는지 인자만 봐도 드러나고 ② 따로 떼어 테스트하기 쉽고 ③ 숨은 의존이 없습니다. (Three.js 객체를 변이하므로 완전한 수학적 순수함수는 아니고, "전역 의존 없이 인자 기반으로 동작"하는 느슨한 의미입니다.)

5 — 한눈에 Before / After
[ Before ]                       [ After ]
buildSim() 2333줄                 simulation.js (조립자, ~250줄)
 ├ 상태 50개 (클로저)      →        └ ctx 하나 생성 후 모듈에 넘김
 ├ makeLed()                       sim/context.js  ← 상태 컨테이너
 ├ setGunFire()                    sim/leds.js     ← setEye(ctx,...)
 ├ setRocketLaunch()               sim/gun.js      ← setGunFire(ctx)
 └ … (전부 한 함수 안)             sim/rocket.js, movement.js … (총 14개)
비유: 지금은 50명이 한 방에서 책상 없이 바닥에 물건을 늘어놓고 일하는 상태입니다(누가 뭘 쓰는지 알 수 없고, 방을 나누면 물건이 흩어짐). ctx는 공용 캐비닛 하나를 두고, 작업자(모듈)들이 각자 방으로 흩어지되 필요할 때 같은 캐비닛을 열어 쓰는 방식입니다.

위험도가 "높음"인 이유: 50개 변수를 어느 모듈에 둘지 정확히 분류해 ctx 구조를 설계하고, 함수 시그니처를 전부 (ctx, ...)로 바꿔야 해서 손이 많이 갑니다. 그래서 로드맵 맨 마지막(6단계)에 둡니다.

유형 B 상세설명 — "거대 디스패처 + 핸들러 더미"의 의미

대표 사례는 펌웨어의 process_data.py입니다. 명령 문자열을 받아 어떤 하드웨어를 움직일지 고르는 process() 한 메서드가 170줄이고, 그 뒤에 30개 핸들러가 같은 클래스에 줄지어 있습니다.

1 — 지금 생긴 모양

하나의 거대한 if / elif 사슬이 50개 명령을 전부 분기합니다.

class CommandProcessor:
    def process(self, cmd):                 # ← 이 메서드 혼자 170줄
        if cmd == "PING":         return self._handle_ping()
        elif cmd == "STOP_ALL":   return self._handle_stop_all()
        elif cmd.startswith("SERVO_t"):  return self._handle_timed_wheel(cmd)
        elif cmd.startswith("SERVO_"):   return self._handle_continuous_wheel(cmd)
        elif cmd.startswith("DC_t"):     return self._handle_timed_dcmotor(cmd)
        elif cmd.startswith("LED_ON"):   return self._handle_led_on(cmd)
        elif cmd.startswith("MSG"):      return self._handle_msg(cmd)
        # … 이런 분기가 130개 …
        elif cmd.startswith("GUN_FIRE"): return self._handle_gun_fire()

    def _handle_timed_wheel(self, cmd): ...   # 30개 핸들러가
    def _handle_led_on(self, cmd): ...        # 전부 같은 클래스 안에
    def _handle_msg(self, cmd): ...           # 줄줄이 늘어섬

문제: 새 명령을 추가할 때마다 170줄짜리 분기문 한가운데를 건드려야 하고, 서보·LED·OLED·총 등 전혀 다른 도메인의 코드가 한 파일에 섞여 있어 한 곳을 고치다 다른 도메인을 깨뜨리기 쉽습니다.

2 — 유형 A와 무엇이 다른가

유형 A는 상태가 엉킨 문제였다면, 유형 B는 분기(라우팅) 로직이 한 덩어리인 문제입니다. 다행히 핸들러들은 이미 _handle_xxx로 잘 나뉘어 있어, "고르는 부분"과 "실행하는 부분"을 떼어내기만 하면 됩니다.

3 — 대응책: dict 디스패치 + 도메인 핸들러 모듈

거대한 if/elif를 이름 → 함수 표(dict)로 바꾸고, 핸들러를 도메인별 파일로 옮깁니다.

# handlers/display.py — OLED 도메인만 모음
def handle_msg(robot, cmd): ...
def handle_clear_rect(robot, cmd): ...

# process_data.py — 이제 "라우팅"만 (170줄 → ~30줄)
from handlers import movement, leds, display, system

ROUTES = {                          # prefix → 핸들러 표
    "SERVO_t": movement.handle_timed_wheel,
    "LED_ON":  leds.handle_led_on,
    "MSG":     display.handle_msg,
    # …
}

def process(self, cmd):
    for prefix, handler in ROUTES.items():
        if cmd.startswith(prefix):
            return handler(robot, cmd)   # ✅ 표에서 찾아 위임

이제 새 명령은 해당 도메인 파일에 함수 하나 추가 + 표에 한 줄 등록으로 끝납니다. 거대 분기문을 건드릴 필요가 없습니다.

4 — 한눈에 Before / After
[ Before ]                       [ After ]
process_data.py 653줄             process_data.py (라우팅, ~30줄 + 표)
 ├ process() 170줄 (if/elif)  →    handlers/movement.py
 ├ _handle_timed_wheel()           handlers/dcmotor.py
 ├ _handle_led_on()                handlers/leds.py
 ├ _handle_msg()                   handlers/display.py
 └ … 30개 핸들러 (한 클래스)      handlers/system.py · buzzer.py · sensors.py
주의 — MicroPython RAM 제약: Pico에서는 파일을 잘게 쪼갤수록 import 오버헤드(메모리)가 늘어납니다. 그래서 도메인을 5~6개 수준으로만 묶고 과분할은 피합니다. (웹의 generateCommand() switch도 같은 패턴이지만 브라우저라 이 제약은 없습니다.)

유형 C 상세설명 — "데이터 덩어리 / 만능 함수"의 의미

대표 사례는 blocklyconfig.js입니다. 블록 21종의 정의가 한 배열에 385줄로 들어 있습니다. 로직이 아니라 나열된 데이터가 길어서 큰 경우입니다.

1 — 지금 생긴 모양
export const BlocklyConfig = {
    blocks: [
        // ── 서보 모터 (9개) ──
        { type: "timed_forward",  message0: "...", colour: "#FF8C00", ... },
        { type: "move_forward",   message0: "...", colour: "#FF8C00", ... },
        // ── DC 모터 (5개) ──
        { type: "main_motor_forward_timed", ... },
        // ── LED (4) · 디스플레이 (5) · 부저 (2) · 총 (1) · 센서 (3) … ──
        // … 한 배열 안에 21종이 385줄로 쭉 ……
    ]
};

이미 주석으로 카테고리가 갈려 있다는 점이 핵심입니다. 분기 로직이 아니라 데이터라서, 경계가 이미 또렷합니다. (ai_helper.js의 matchAction() 149줄도 같은 부류 — 한 함수가 이동·센서·디스플레이·소리 의도를 카테고리별 if로 나눠 처리하고 있어 카테고리 경계가 분명합니다.)

2 — 유형 B와 무엇이 다른가

유형 B는 "고르는 로직"을 떼어내는 작업이었습니다. 유형 C는 떼어낼 로직이랄 게 없고, 이미 그어진 카테고리 선을 따라 파일만 물리적으로 나누면 됩니다. 그래서 가장 기계적이고 안전합니다.

3 — 대응책: 카테고리별 파일 + index.js 재조립
// blocks/servo.js — 서보 블록만
export const servoBlocks = [
    { type: "timed_forward", ... },
    { type: "move_forward",  ... },
];

// blocks/led.js, blocks/display.js … 각 카테고리 1파일

// blocks/index.js — 흩어진 카테고리를 다시 합쳐 원래 형태로
import { servoBlocks }   from './servo.js';
import { dcmotorBlocks } from './dcmotor.js';
import { ledBlocks }     from './led.js';
// …
export const BlocklyConfig = {
    blocks: [ ...servoBlocks, ...dcmotorBlocks, ...ledBlocks, ... ]
};

외부에서 쓰는 코드는 여전히 BlocklyConfig.blocks를 그대로 받으므로 나머지 코드는 한 줄도 바꿀 필요가 없습니다. 새 블록은 해당 카테고리 파일에만 추가하면 됩니다.

4 — 한눈에 Before / After
[ Before ]                       [ After ]
blocklyconfig.js 416줄            blocks/servo.js   (9종)
 └ blocks: [ 21종 385줄 ]    →    blocks/dcmotor.js (5종)
   (주석으로만 구분)              blocks/led.js · display.js · buzzer.js …
                                  blocks/index.js  ← 합쳐서 BlocklyConfig 재구성
핵심: 유형 C는 "분할"이라기보다 이미 있는 구분선을 파일 경계로 승격시키는 작업입니다. 동작이 1:1로 보존되고 위험이 낮아, 로드맵에서 이른 단계(2단계)에 둡니다.

유형 D 상세설명 — "인라인 자산"의 의미

대표 사례는 dashboard.html입니다. 한 파일 828줄인데, 그중 CSS가 362줄·JavaScript가 385줄로 HTML 태그 안에 직접 박혀 있습니다("인라인").

1 — 지금 생긴 모양
<!-- dashboard.html — 한 파일에 세 언어가 섞임 -->
<head>
    <style>
        /* … CSS 362줄 … */
    </style>
</head>
<body>
    <!-- HTML 마크업 187줄 -->
    <script>
        // … JavaScript 385줄 …
    </script>
</body>

문제: 구조(HTML)·표현(CSS)·동작(JS)이 한 파일에 섞여 있어 ① 무엇을 고치든 828줄을 헤집어야 하고 ② 브라우저가 CSS/JS를 따로 캐싱하지 못하며 ③ 여러 명이 동시에 작업하면 충돌이 잦습니다.

2 — 다른 유형과 무엇이 다른가

A·B·C는 "한 언어 안에서 너무 큰 단위"를 나누는 문제였습니다. 유형 D는 아예 성격이 다른 세 가지가 한 파일에 동거하는 문제라, 언어 경계를 따라 떼어내기만 하면 됩니다. 가장 쉽고 위험이 가장 낮습니다.

3 — 대응책: CSS/JS 외부 파일로 추출
<!-- dashboard.html — 순수 마크업 187줄만 남음 -->
<head>
    <link rel="stylesheet" href="dashboard.css">   <!-- CSS 분리 -->
</head>
<body>
    <!-- HTML 마크업 -->
    <script src="dashboard.js"></script>        <!-- JS 분리 -->
</body>

/* dashboard.css */  ← <style> 내용을 그대로 이동 (362줄)
// dashboard.js     ← <script> 내용을 그대로 이동 (385줄)

내용을 그대로 잘라 옮기고 <link>·<script src>로 연결만 하므로 동작이 완전히 동일합니다. main.html의 인라인도 같은 방식으로 처리합니다.

4 — 한눈에 Before / After

05 — Cross-cutting Concerns

횡단 주의점

분할이 깨뜨리기 쉬운 결합 지점들입니다.

  • 블록 타입 이름 3중 결합 — blocklyconfig.js의 type ↔ commandexecutor.generateCommand()의 case ↔ ai_helper.KNOWN_TYPES. 한곳(상수)으로 모아 단일 출처(SSOT)화하면 동기화 버그를 줄입니다.
  • state.js 전역 의존 — 거의 모든 Web 모듈이 state를 직접 변이. 분할 시 누가 무엇을 쓰는지 드러나므로 이 기회에 접근 경로를 정리합니다.
  • firmware ↔ web 문자열 프로토콜 — 컴파일 검증이 안 됨. Document/API_DOCUMENTATION.md가 유일한 계약 → 변경 시 동시 갱신.
  • 알려진 버그 동거 — wheel turn 로직, SYS_SET 고아 코드 등 기존 버그는 분할과 분리해서 다룹니다. 리팩터링 커밋에 버그 수정을 섞으면 회귀 추적이 어려워집니다.
  • MicroPython RAM 제약 — Pico에서는 파일 분할 시 import 오버헤드가 늘어납니다. 도메인 5~6개 수준으로만 나누고 과분할은 피합니다.

1차 유형 A(로드맵 6단계)를 2차의 본격 착수 항목으로 승격합니다.

1.1요소별 코드 분할

Web/simulation.js(약 2,333줄)의 거대 클로저를 요소 단위(로버 부품·발사대·신호등·LED·총·로켓·음파·OLED)로 분할합니다. 1차 유형 A의 sim/ 서브시스템(context·assets·leds·movement·gun·rocket·traffic·waves·oled·render·audio·dispatch·topics)을 실제로 도입해, 한 요소를 독립적으로 보고 고칠 수 있게 합니다.

↳ 1차 로드맵 6단계(simulation.js → sim/)와 동일 작업. 2차에서 본격 착수합니다.

1.2단위 메쉬 로딩 후 개발자 도구로 Transform 조작

부품 메쉬(Web/Mesh/RoverParts/*.glb, EnvAssets/*)를 개별(단위) 로딩한 뒤, 브라우저 개발자 도구로 위치·회전·스케일(Transform)을 실시간 조작해 정렬값을 잡고, 확정값을 코드(상수/설정)로 역반영합니다. 하드코딩 좌표를 눈대중으로 맞추던 수작업을 도구화합니다.

  • 구현 아이디어 — 디버그 모드(?debug=1)에서 three.js TransformControls 부착 → 선택 객체의 position/rotation/scale을 패널·콘솔에 출력 → 코드로 붙여넣기.
  • 단위 메쉬 로더 — 부품을 하나씩 로딩·점검하는 경로로 모델 교체·정렬 검증을 빠르게.

  • 텍스트 리빌 — 라인 단위로 아래→위 등장. 1.2s cubic-bezier(0.25,1,0.5,1), translateY(1.3em) → 0.
  • Split Heading — 단어별 순차(stagger) 등장 · 문자 단위 리빌 옵션.
  • 스크롤 연동 — 뷰포트 진입 시 페이드인(wpb_animate_when_almost_visible 패턴).
  • 호버 — 이미지 brightness(80%) · 버튼 배경 #383838.
  • 로딩 — 스피너 rotate 0.6s linear infinite · 카운트 펄스 1s.
