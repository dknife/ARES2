1. 개요

  • 본 문서는 ARES 교육 플랫폼의 연결·통신·블록엔진·액추에이터·센서·설정·안정성·시뮬레이터·E2E 전 영역에 대한 1차 테스트 수행 결과를 정리함
  • 테스트는 실제 UART 명령 프로토콜과 블록-명령 매핑에 대응되는 TCL v1.0 98 케이스를 기준으로 하였으며, 하드웨어 의존 항목은 실기(BLE)로, 그 외는 Web/Sim_Parts 시뮬레이터로 병행 검증함
  • v1.0 문서는 소스코드 분석 기반의 예상치였으나, 본 v1.1은 테스터가 TCL 실행형 시트에 기록·내보내기한 실측 결과와 미결 13건의 코멘트(r.txt)를 취합하여 실제 수치로 전면 갱신함
  • 미결 13건은 모두 Fail(기능 결함)이 아닌 검증 불가·환경 부족·명세 문제로 확인되어, 결함 관리와 별도로 테스트 가능성(Testability) 개선 항목으로 관리함

5.1 유형 A : Web 수신 로그 확인 불가 (보류 5건)

  • 공통 현상 : 5건 모두 테스터가 "작동으로 보이지만, 현재 Web 로그 출력 방식상 출력을 확인할 수 없습니다"로 동일하게 기록 - 물리 동작(OLED 표시, 순차 실행, 멜로디 재생)은 관찰되나 합격 기준의 응답 문자열(1, READY) 대조가 불가
  • 원인 분석 : Web 로그가 송신(tx) 명령 중심으로 출력되고 bluetooth.js notification 수신부의 수신(rx) 문자열을 사용자 로그에 노출하지 않음 - 시뮬레이터 경로는 → 송신 / ↩ 응답을 모두 표기(TC-SIM-004 Pass)하는 것과 대조적으로 실기 BLE 경로에만 수신 표시가 없음
  • 간접 증거 : 응답 대기형 명령(SLEEP·SERVO_t*·BATCH·SING)이 RESPONSE_TIMEOUT(5000ms) 오류 없이 다음 블록으로 진행된 점은 응답이 정상 수신되고 있음을 간접적으로 뒷받침 - 단 문자열 일치·응답 횟수(1회) 검증은 rx 로그 없이는 불가하므로 Pass 판정을 유보함
  • 공통 조치 : 실기 BLE 경로에도 수신 문자열을 로그에 표시하는 기능 추가(IMP-001, High) 후 5건 일괄 재시험 - 재시험 시 Pass 전환 유력

순번 1 · TC-CONN-001 연결 확인 PING High보류

작동으로 보이지만, 현재 Web 로그 출력 방식상 출력을 확인할 수 없습니다
  • 합격 기준 : PING 전송 시 OLED에 CONNECTED! + 기기이름 표시, 응답 1 수신
  • 분석 : OLED 표시(물리 관찰)는 확인 가능한 반면 응답 1의 수신 여부를 Web 화면에서 확인할 수 없음 - 합격 기준이 두 요소(표시 + 응답)의 AND 조건이므로 보류 처리가 타당
  • 조치 : IMP-001 반영 후 재시험 - OLED 표시가 이미 관찰되었다면 _handle_ping 경로 자체는 실행된 것이므로 실질 위험은 낮음

순번 2 · TC-CONN-002 READY 응답 Low보류

작동으로 보이지만, 현재 Web 로그 출력 방식상 출력을 확인할 수 없습니다
  • 합격 기준 : READY 전송 시 READY 문자열 그대로 회신
  • 분석 : 물리적 부수 효과가 전혀 없는 순수 에코 케이스로, rx 로그 없이는 검증 수단이 원천적으로 부재한 대표 사례 - 유형 A 중 재시험 의존도가 가장 높음
  • 조치 : IMP-001 반영 후 재시험 - 우선순위 Low로 배포 판단에 미치는 영향은 제한적

순번 9 · TC-BLE-002 페어링·특성 확보 High보류

작동으로 보이지만, 현재 Web 로그 출력 방식상 출력을 확인할 수 없습니다
  • 합격 기준 : 기기 선택 후 특성(0000ffe1) 확보, notification 구독, 상태 아이콘 초록 전환
  • 분석 : 상태 아이콘 초록 전환은 UI로 관찰 가능하며 이후 명령 송수신이 성립했으므로 GATT 특성 확보·구독 자체는 성공한 것으로 판단됨 - 다만 특성 UUID·구독 성공 로그를 화면에서 확인할 수 없어 명세 문구 기준으로는 보류 기록
  • 조치 : IMP-001에 연결 수립 단계 로그(특성 확보·구독 성공) 표시를 포함하여 재시험 - 재시험 전이라도 후속 케이스 전반의 Pass가 본 케이스의 실질 성공을 강하게 뒷받침

순번 17 · TC-BAT-002 순차 실행·단일 응답 High보류

작동으로 보이지만, 현재 Web 로그 출력 방식상 출력을 확인할 수 없습니다
  • 합격 기준 : BATCH;LED_ON,0,1|LED_OFF,0의 두 명령 순차 실행 후 응답 1을 한 번만 수신
  • 분석 : LED 점등·소등의 순차 실행은 육안으로 확인 가능하나, "응답 1회"라는 횟수 검증은 rx 로그가 없으면 불가 - BATCH 응답이 중복되면 응답 슬롯 오염으로 후속 blocking 명령이 오동작할 수 있어 High 유지가 타당
  • 조치 : IMP-001 반영 후 재시험 - 재시험 시 응답 횟수까지 로그로 계수 확인. 참고로 BATCH 직렬화(TC-BAT-001)·금지 블록 차단(TC-BAT-003~005)·부저 드레인(TC-BAT-006)은 모두 Pass로 BATCH 계열의 잔여 위험은 본 건 하나로 수렴

순번 54 · TC-SND-003 멜로디 SING Low보류

작동으로 보이지만, 현재 Web 로그 출력 방식상 출력을 확인할 수 없습니다
  • 합격 기준 : SING 시 Hala Madrid 멜로디 재생, OLED Singing... 표시, 응답 1
  • 분석 : 멜로디 재생과 OLED 표시는 관찰되었으나 blocking 응답 1의 수신 확인 불가 - 멜로디 종료 후 다음 블록 진행이 timeout 없이 이루어졌다면 응답은 정상 수신된 것으로 추정
  • 조치 : IMP-001 반영 후 재시험

5.2 유형 B : 장비·환경 부족 (보류 1건 + 미수행 3건)

순번 8 · TC-BLE-001 기기 검색·필터 High보류

이름은 분류 여부도 확실치 않고, 기기가 여러개 있는 상황등, 테스트 할 제품이 부족합니다

  • 시뮬레이터 자기 센서 미구현 : Web/Sim_Parts의 MAGNET 응답은 항상 MAG:0으로 고정되어 자기 감지 시나리오는 시뮬레이터로 검증 불가 - TC-SEN-003 실기 테스트로 커버(이번 회차 Pass), 시뮬레이터 자기 감지 구현은 후속 과제 유지

11. 최종 평가 및 결론

  • 긍정 근거 : 실행 85건 전건 Pass·Fail 0건, High 이상 신규 결함 0건, 하드웨어 구동(서보·DC·LED·OLED·센서)과 시뮬레이터 전 경로(28건) 및 프리셋 E2E 정상 완주, 비상 정지·None 가드·설정 영구화 등 안전·안정성 항목 전건 통과
  • 유보 근거 : High 우선순위 케이스 5건(TC-CONN-001, TC-BLE-001·002·006, TC-BAT-002)이 보류 상태이며, 보류 원인인 rx 로그 기능(IMP-001)이 미반영 - 응답 프로토콜의 문자열 수준 검증과 KPI 정밀 실측이 미완
  • 판정 : 미결 13건이 모두 제품 결함이 아닌 검증 수단·환경·명세 문제로 분석되므로 기능 품질 자체는 배포 수준으로 평가하되, High 보류의 독립 검증이 남아 있어 조건부 Go로 판정함
  • 확정 조건 : ① IMP-001 반영 후 재시험 R1 완료 ② R2(절차 보강 재시험) 완료 ③ 총 모듈 의존 R3와 테스트 베드 의존 R4는 현장 실증 단계 병행 허용 - ①·② 완료 시 Go 확정

판정 : 조건부 Go (재시험 R1·R2 완료 후 확정)   승인 : 품질관리(QA) 책임자
