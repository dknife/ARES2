2. 테스트 산출물 상세 가이드 및 ARES 실 매핑

1. STP (Software Test Plan) - 테스트 계획서

요구사항 분석 단계

역할과 책임 (R&R)

  • 작성자 : PM 또는 QA Lead
  • 검토/승인 : 발주처, 이해관계자
  • 역할 : 예산·일정·범위 정의 및 테스트 전략 확립

작성 핵심 포인트

  • 교육 현장 통신 안정성(BLE) 중점
  • 시뮬레이터 선행 테스트 전략 포함
  • 실제 명령 프로토콜 기반 KPI 정의

ARES 적용 사례

  • 범위(In-Scope) : Web 블록엔진, BLE 파이프라인, Pico 명령 처리, 액추에이터(서보/DC/LED 6개/부저/발사), 센서(초음파/디지털 자기), OLED, 설정 영구 저장
  • KPI 정의 : cooldown 40ms, 처리량 100 cmd/s, RESPONSE_TIMEOUT 5초 내 응답, DC 양방향 대칭 (모두 실제 코드 설정값 기준)

문서 템플릿 요약

개요목적, 대상 (ARES 웹 IDE 및 Pico 펌웨어)
테스트 범위블록-명령 변환, BLE 통신, 모터/센서/LED/OLED 제어, BATCH 블록, 설정 저장
테스트 환경Chrome/Edge, Windows/Android, ARES 하드웨어, Web/Sim_Parts 시뮬레이터
KPI명령 간격, 처리량, 모터 대칭, 페어링 시간
리스크BLE 혼선, 브라우저 호환, 부팅 모듈 초기화, BATCH SLEEP timeout, UART baud 천장

2. TDS (Test Design Specification) - 테스트 설계서

시스템 설계 단계

역할과 책임 (R&R)

  • 작성자 : QA Architect 또는 Tech Lead
  • 검토/승인 : PM, 개발 팀장
  • 역할 : 테스트 대상을 논리 그룹으로 묶고 합격 기준 수립

작성 핵심 포인트

  • 정상 동작 외 오류 복구·경계값 설계
  • 통신 단절 재연결 설계
  • 응답 정책(무응답 vs 응답 대기) 분류 검증 설계

ARES 적용 사례 - 응답 정책 동기화 설계

  • 가설 : Web FIRE_AND_FORGET_HEADS 와 Pico NO_RESPONSE_CMDS 가 불일치하면, 무응답 명령의 ack가 다음 명령 응답 슬롯을 오염시킴
  • 설계 기법 : 두 집합의 원소를 대조하는 정합성 검사 + LED_ON 직후 DISTANCE 시퀀스로 오염 여부 관찰
  • Pass 기준 : 두 집합 동일, DISTANCE 값이 LED ack로 오염되지 않음 (TC-BLE-004~006)

경계값 분석 사례 - 모터 속도

  • DC 속도 입력 -1, 0, 100, 150 대비 : 0~100 clamp 후 PWM 매핑 확인 (TC-DCM-005)
  • 서보 속도가 max_speed 상한을 넘지 않는지 (TC-SRV-005)

문서 템플릿 요약

기능 모듈명블록 변환 및 전송 엔진 (Block → 명령 문자열 → BLE)
테스트 목적generateCommand의 정확한 명령 생성 및 보드 실행 확인
적용 기법구조 기반(블록 조합), 명세 기반(응답 문자열), 응답 정책 분류, 부하(연속 100 cmd)
Pass/Fail 기준(Pass) 명령 문자열 정확 생성 + 보드 실행 / (Fail) 누락·지연·응답 오염

3. TCL (Test Case List) - 테스트 케이스 명세서

상세 설계 / 구현 단계

역할과 책임 (R&R)

  • 작성자 : QA Engineer, 개발자
  • 검토/승인 : QA Lead
  • 역할 : 실행 가능한 시나리오 단위 리스트 도출

작성 핵심 포인트

  • 입력/예상 결과를 실제 명령·응답 문자열로 기재
  • Positive 외 예외(미장착·범위초과) 케이스 도출
  • 비고 칸에 실제 소스 함수/파일 인용

ARES 적용 사례 - TCL v1.0 (14 카테고리 98 케이스)

  • TC-LED-001 : 6개 LED(GP21,20,19,18,17,16) 인덱스 0~5 점등 - 입력 LED_ON,0,1~LED_ON,5,1, 비고 leds.py
  • TC-SEN-003 : 디지털 자기장 - 입력 MAGNET, 예상 MAG:1/MAG:0 (active-low, ADC 아님), 비고 magsensor.py
  • TC-DCM-002 : dual PWM 양방향 대칭 - DC_FORWARD/BACKWARD 80% 이동거리 근사 대칭, 비고 dcmotor.py

4. STD (Software Test Description) - 테스트 절차서

테스트 실행 전

역할과 책임 (R&R)

  • 작성자 : QA Engineer
  • 사용자 : 실제 테스터, 교사
  • 역할 : TC를 비전문가도 따라 할 수 있게 매뉴얼화

작성 핵심 포인트

  • 재현 가능성 확보
  • 클릭 위치·입력값·확인 화면 상세 기술
  • 실기와 시뮬레이터 경로를 모두 안내

ARES 적용 사례 - Test All Hardware.xml 실행 절차

  1. 로버 키트 전원 ON (부팅음 G5→C4, OLED "ARES READY" 확인)
  2. Chrome에서 Web/index.html 접속 후 관제실 진입, "블루투스 연결"로 ARES 페어링
  3. 예제/불러오기로 XML Presets/Test All Hardware.xml 로드
  4. "테스트 실행" 함수 호출 - 서보→DC→LED→디스플레이→소리→전투→센서→시간→제어 순차 구동
  5. 각 모듈의 물리 동작을 육안 확인하고 TCL의 TC-E2E-001 결과 기입

하드웨어가 없으면 Web/Sim_Parts 시뮬레이터로 동일 블록을 실행해 명령 로그를 확인 (TC-E2E-003), 단 MAGNET 응답은 시뮬레이터에서 항상 MAG:0으로 고정됨

단일 블록 절차 예 - LED 점등

  1. 미션 뷰에서 LED 탭의 "LED 켜기" 블록 드래그
  2. 번호 1, 밝기 1 입력
  3. "명령 실행" 클릭 - LED_ON,1,1 전송 확인
  4. 2번 LED(GP20) 점등 육안 확인

4. 운영 매뉴얼 (강사·교사용) 핵심 가이드

실제 코드 특성상 유의해야 할 운영 지침을 강사/교사 매뉴얼에 포함함

  • 분단별 순차 페어링 : 다수 동시 페어링 시 BLE 혼선 - 기기 이름으로 구분하고 5명씩 순차 연결
  • BATCH 안 SLEEP 누적 4초 이내 : RESPONSE_TIMEOUT 5초 초과 시 timeout으로 오인 - 긴 대기는 BATCH 밖 SLEEP 블록 사용
  • BATCH 금지 블록 : 값반환(거리/자기/PING)·제어흐름(반복/조건/변수/함수)·batch 중첩은 넣을 수 없음
  • 하드웨어 사양 정정 : LED는 6개, 자기센서는 디지털(1/0), 바퀴는 연속회전 서보 (각도 위치 제어 아님)
  • 설정 저장 : SYS_SET은 ModelFactorySetting.txt에 저장되어 재부팅 후에도 유지 - 잘못 저장 시 파일을 메모장으로 수정
  • 수업 전 배터리 완충 점검 : 전압 감시가 미구현(get_battery_voltage()가 항상 100)이라 방전 시 서보·모터가 조용히 약해질 수 있음 - 수업 시작 전 완충 확인 (STP 리스크 6)
  • BLE 끊김 시 안전 : 모터 구동 중 연결이 끊기면 실행이 중단되고 재연결 안내가 뜸 - 로버가 계속 움직이면 즉시 STOP_ALL 또는 Pico 전원 차단 (STP 리스크 7)
  • 시뮬레이터 활용 : 하드웨어가 없거나 학교망에서 BLE가 막히면 Web/Sim_Parts로 블록-명령 경로를 먼저 검증
  • 비상 정지 범위 : STOP_ALL/STOP은 ABORT_CMDS로 분류되어 진행 중인 타임드 동작(SERVO_t*/DC_t*/SLEEP)도 즉시 중단시킴 - 로버가 계속 움직이면 이 두 명령 중 하나로 즉시 정지 가능
