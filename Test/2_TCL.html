SETUP

실행 준비 - 테스트 수행 방법

경로 A : 실제 하드웨어 (BLE 연결)

  1. ARES 로버 키트 전원 ON - Pico가 main.py 부팅 (부저 부팅음 G5→C4, OLED에 "ARES READY" 표시)
  2. Chrome 또는 Edge에서 Web/index.html 접속 후 관제실(main.html) 진입
  3. "블루투스 연결" 클릭 - 서비스 UUID 0000ffe0-... 필터로 ARES 기기 선택 (기본 이름 ARES)
  4. 미션 뷰에서 블록 조립 후 "명령 실행" - 각 블록이 UART 명령으로 전송됨

경로 B : 시뮬레이터 (하드웨어 없이 실행)

  1. Web/Sim_Parts의 시뮬레이터로 블록 실행 - commandexecutor.js의 simulateWorkspace()가 simSink로 명령을 흘려보냄
  2. DISTANCE/MAGNET는 sink 응답을 _parseSensorReply()가 파싱해 _last_distance/_last_magnetic 변수에 반영
  3. 부저 음 길이 페이싱, 반복/조건 흐름 등 Web 측 로직을 하드웨어 없이 검증 가능
COLLAB

다수 인원 분산 테스트 및 결과 취합 방법

1단계 : 카테고리 배정 (예시 - 7명 기준)

14개 카테고리를 7명에게 2개씩 배정하면 겹치지 않게 나눌 수 있음(팀 상황에 맞게 조정 가능)

  1. 테스터 1 : 연결·시스템(7) + BLE·파이프라인(8)
  2. 테스터 2 : BATCH 블록(6) + 블록 코딩 엔진(7)
  3. 테스터 3 : 서보 모터 휠(6) + DC 모터(5)
  4. 테스터 4 : LED 6개(6) + 디스플레이 OLED(6)
  5. 테스터 5 : 소리·부저(5) + 전투(3) + 센서 입력(5)
  6. 테스터 6 : 설정·안정성(6) + 시뮬레이터·프리셋 E2E(4)
  7. 테스터 7 : 시뮬레이터 3D(24)

2단계 : 개별 테스트 및 내보내기

  1. 각자 이 TCL 파일을 자신의 브라우저에서 열고, 하단 결과 집계 영역의 "테스터 이름"에 본인 이름 입력
  2. 배정된 카테고리만 실제 테스트 후 결과 드롭다운을 Pass/Fail/보류/기타로 변경 - 이 브라우저의 localStorage에 자동 저장됨
  3. 테스트가 끝나면 "결과 내보내기 (JSON)" 클릭 - ares_tcl_results_이름_날짜.json 파일이 다운로드됨(미수행 항목은 제외되고 실제 기록된 결과만 담김)
  4. 다운로드된 JSON 파일을 취합 담당자에게 전달(메일/메신저/공유 드라이브)

3단계 : 취합 및 STR 작성

  1. 취합 담당자는 동일한 TCL 파일을 결과가 비어있는 새 브라우저(또는 시크릿 창)에서 엶
  2. "결과 가져오기 (병합)" 클릭 후 7명의 JSON 파일을 한 번에 선택(다중 선택 가능) - 파일별 반영 건수와 충돌 건수가 하단 "가져오기 내역"에 기록됨
  3. 같은 케이스가 서로 다른 값으로 중복 기록된 "충돌"이 있으면 알림으로 표시되므로, 해당 테스터에게 재확인 후 드롭다운을 직접 수정
  4. 모든 파일을 가져오면 상단 실행 결과 집계가 7명분을 합친 최종 수치로 자동 계산됨 - 이 수치를 그대로 ARES_3_STR.html의 "테스트 수행 결과 요약"·"카테고리별 결과" 표에 옮겨 작성

참고 : 내보내기는 미수행이 아닌(실제 기록된) 항목만 포함하므로, 서로 다른 사람이 담당 외 카테고리를 건드리지 않았다면 가져오기 시 다른 사람의 결과를 덮어쓸 위험이 없음

E2E

XML 프리셋 실행 절차 (Test All Hardware / Launch Rocket)

절차 1 : 하드웨어 전수 테스트 (Test All Hardware.xml)

프리셋의 "테스트 실행" 함수는 9개 하위 함수를 순차 호출함. 각 함수가 생성하는 실제 명령은 다음과 같음

  1. 서브 모터 : SERVO_tFORWARD,1 → tBACKWARD → tLEFT → tRIGHT → 연속 FORWARD/STOP/BACKWARD/STOP/LEFT/STOP/RIGHT/STOP
  2. DC 모터 : DC_tFORWARD,1 → tBACKWARD → 연속 FORWARD/STOP/BACKWARD/STOP
  3. LED : LED_ON,1~5 순차 → LED_OFF,ALL → 패턴 [1 1 1 1 1] → 소등
  4. 디스플레이 : MSG,Hello → CLEAR_DISPLAY
  5. 소리 : BUZZER_ON,262,1 → BUZZER_ON,392,1
  6. 전투 : GUN_FIRE
  7. 센서 : DISTANCE → 결과 MSG 표시 → MAGNET → 결과 MSG 표시
  8. 시간 : 부저 → SLEEP,1 → 부저 → SLEEP,1
  9. 제어·변수·수학 : 반복(GUN_FIRE 3회) + while/until 루프 + 랜덤 + 조건 분기

합격 기준 : 9개 함수 전부 오류 로그 없이 순차 실행, 각 모듈의 물리 동작 육안 확인

절차 2 : 발사 시퀀스 (Launch Rocket.xml)

  1. 변수 LED Index=5 설정
  2. 5회 반복 : LED_ON,(Index),1 → SLEEP,1 → BUZZER_ON,262,1 → Index=Index-1
  3. 반복 종료 후 GUN_FIRE
  4. LED_OFF,ALL

합격 기준 : LED가 5→1 카운트다운 점등, 매 단계 부저음, 마지막 발사 후 전체 소등

NOTE

수행 시 유의 사항

  • LED는 6개임 (GP21,20,19,18,17,16 / 인덱스 0~5) - 블록/명령은 0~5로 clamp
  • 자기장 센서는 디지털 홀/리드 스위치(active-low)로 1/0만 반환 - 아날로그 ADC 아님
  • 바퀴 서보는 연속회전 서보 - 전진/후진/선회로 제어하며 각도 위치 제어가 아님
  • 부저 BUZZER_ON은 논블로킹 - 음 길이 페이싱은 Web이 담당, 자동 정지는 메인 루프 update()가 담당
  • BATCH 안에는 값반환(거리/자기/PING)·제어흐름(반복/조건/변수/함수)·batch 중첩 금지
  • 기본 설정 : max_speed=40, collision_dist=10, auto_stop=1, device_name=ARES (ModelFactorySetting.txt)
  • STOP_ALL / STOP(레거시)은 ABORT_CMDS에 속해, SERVO_tFORWARD/DC_tFORWARD/SLEEP처럼 진행 중인 타임드(블로킹) 동작을 카운트다운 도중이라도 즉시 중단시킴 - TC-CONN-007에서 이 중단 거동을 함께 확인할 것
  • GUN_FIRE는 다른 핸들러와 달리 gun 모듈이 없어도(None) 항상 1을 응답함 - 무응답(fire-and-forget)이라 Web에서 체감되진 않지만 하드웨어 유무 진단 목적으로는 사용 불가
  • 시뮬레이터의 MAGNET 응답은 항상 MAG:0 고정(실제 자기 감지 미구현) - 자기 센서 시나리오는 실기(TC-SEN-003)로만 검증 가능하며, DISTANCE만 raycast 기반 실측이 반영됨
  • 3D 시뮬레이터의 실제 구현 위치는 Web/Sim_Parts/(dispatch·movement·leds·oled·gun·rocket·traffic 등)이며, Web/Simulation/Simulation_*.js는 어디에서도 import되지 않는 미사용 파일이므로 코드 추적 시 참조하지 말 것
  • BLE 기기 필터는 서비스 UUID(0000ffe0) 매칭 또는 이름이 PicoBLE/HMSoft/BT05인 경우 모두 노출됨(OR 조건) - "ARES"라는 이름 자체는 필터 목록에 없으며, 부팅 시 _sync_bt_name()이 모듈 광고 이름을 device_name(기본 "ARES")으로 재설정하려 시도하므로 최초 페어링과 재부팅 후 표시 이름이 다를 수 있음
