GitHub Issues 버그 트래킹 구축 가이드

본 가이드는 GitHub Issues와 Pull Request(PR)의 유기적 연동을 통해 버그의 발생부터 수정, 배포까지의 생명주기(Lifecycle)를 추적 관리하는 프로세스를 정의합니다.

1. 버그 리포트 템플릿 자동화 (Issue Template)

일관된 데이터 수집을 위해 저장소 내에 마크다운 기반의 이슈 템플릿을 사전 배치합니다.

경로: .github/ISSUE_TEMPLATE/bug_report.md 파일을 생성하고 아래 내용을 커밋합니다.

---
name: 🐛 Bug Report
about: 오류 보고 및 재현 경로 정의
title: '[모듈명] 오류 현상 요약'
labels: ['bug', 'triage']
assignees: ''
---

### 2. Steps to Reproduce

2. 실무 시나리오 기반 작성 예시

시나리오: ARES 제어 시스템에서 'N번 반복' 명령을 처리할 때, 내부 루프의 경계 조건 오류(Off-by-One)로 인해 명령어가 정확히 N회가 아닌 N-1회만 실행되고 종료되는 버그 발생.

[ARES] 'N번 반복' 처리 시 내부 명령어가 N-1회만 루프를 도는 현상 (Off-by-One) #104

Labels: bug P1-High | Milestone: v1.2.0-rc1

1. Environment

  • Runtime / OS: MicroPython v1.22 / Raspberry Pi Pico (ARES Core)
  • Dependency Version: ares-firmware v1.1.5, servo-controller v0.8.2

2. Steps to Reproduce

  1. ARES 제어 대시보드 또는 JSON 메시지를 통해 {'command': 'repeat_loop', 'count': 5, 'action': 'SERVO_tFORWARD'} 명령을 전송.
  2. 로봇의 펌웨어 디버그 콘솔 출력 또는 OLED 화면의 카운터 값을 확인.
  3. 지정한 5회가 아닌 4회만 전진 명령이 실행되고 루프가 종료됨 (count: 1 지정 시 루프가 아예 실행되지 않음).

3. Expected Behavior

  • N번 반복 설정 시 내부 제어 명령어가 정확히 N회 루프를 돌며 실행되어야 함.
  • 시스템 디버그 로그의 iteration 카운터가 1/N부터 N/N까지 누락 없이 기록되고, total_executed=N으로 완료되어야 함.

4. Actual Behavior / Error Logs

[ARES-System] Received loop command: count=5
[ARES-Loop] iteration 1/5 ... task success
[ARES-Loop] iteration 2/5 ... task success
[ARES-Loop] iteration 3/5 ... task success
[ARES-Loop] iteration 4/5 ... task success
[ARES-Loop] loop completed: total_executed=4 (expected: 5)

5. Context & Speculation

펌웨어 내부의 반복문 조건식이 for i in range(1, count):와 같이 종료 인덱스가 미만으로 처리되었거나, while 문 진입 직후 카운터를 먼저 감소시켜 1회가 유실되는 것으로 추정됨. 경계 조건(N=1, N=5)에 대한 단위 테스트 케이스를 추가하여 실제 실행 횟수가 설정값 N과 일치하는지 검증 필요.

3. 메타데이터 매핑 및 칸반(Projects) 연동

이슈 우측 사이드바를 통해 프로젝트 관리 상태를 갱신합니다.

  • Assignees: 담당 개발자(들)를 지정하여 R&R을 명확히 함.
  • Labels: bug(유형), P1-High(우선순위) 조합으로 필터링 효율화.
  • Milestone: 해당 버그가 릴리스되어야 할 스프린트 혹은 배포 버전에 매핑.
  • Projects: GitHub Projects(Kanban)에 연동하여 Todo -> In Progress -> Done 상태 추적.

4. 개발 수명 주기(Lifecycle)와 자동화 연동

GitHub의 특정 키워드를 활용하면 브랜치 머지 시 이슈를 자동으로 Close 처리할 수 있어, 수동 관리 오버헤드를 줄입니다.

4.1 Branch Naming 규칙 권장

git checkout -b fix/104-repeat-loop-off-by-one

4.2 PR(Pull Request) 또는 Commit Message 연동

코드를 수정하고 main 또는 develop 브랜치로 Pull Request를 보낼 때, 본문에 아래의 **원자적 키워드**를 포함합니다.

## Description
'N번 반복' 명령 처리 루프의 경계 조건(Off-by-One) 수정 및 지정 횟수(N)와
실제 루프 실행 횟수 일치 여부를 검증하는 테스트 추가.

Closes #104
💡 핵심 연동 키워드 (Case-Insensitive)
close #이슈번호, closes #이슈번호, fix #이슈번호, fixes #이슈번호, resolve #이슈번호
해당 PR이 타겟 브랜치에 Merge되는 순간 관련 이슈가 자동으로 Close 처리되며 Traceability 링크가 보존됩니다.
