1. 목적 (Purpose)

본 테스트 계획서(STP)는 "블록코딩과 BLE 통신을 이용한 프로그래밍형 마이크로컨트롤러 기반 교육 플랫폼 (ARES)"의 품질을 검증하기 위한 지침서임

  • Web 블록코딩 실행 엔진의 블록-명령 변환 정확성 검증
  • Web Bluetooth API를 통한 BLE 통신 안정성 및 파이프라인 정합성 검증
  • Raspberry Pi Pico 펌웨어의 명령 처리·하드웨어 제어 정밀성 검증
  • 본 계획은 실제 소스코드(Pico/ 13개 모듈, Web/ 실행 엔진, XML Presets 2종)를 전수 분석하여 실제 명령 프로토콜에 대응되는 테스트 항목만 채택함

본 STP가 근거로 삼는 실제 소스 구조

  • Pico/ : main.py(UART 루프), process_data.py(명령 디스패처), hardware.py(모듈 싱글톤), system_config.py(설정), wheel/dcmotor/leds/buzzer/gun/ultrasonic/magsensor/ssd1306/pins
  • Web/ : commandexecutor.js(블록→명령), bluetooth.js(BLE), blocklyconfig.js(블록 정의·BATCH 금지), simulation.js·Sim_Parts/(시뮬레이터 실 구현체), main.js(12차시 셸)
  • XML Presets/ : Test All Hardware.xml(전 모듈 전수 테스트), Launch Rocket.xml(LED 카운트다운 발사 시퀀스)
  • examples/ : countdown, greeting, led_variable, obstacle_avoid, radar_demo, rover_wakeup, albi_song (7종 예제)

2. 테스트 범위 (Test Scope)

범위는 Web UI와 Pico 펌웨어 간의 연동을 포함한 전체 시스템 융합 테스트임

  • 연결·시스템 조회 : PING, READY, GET_STATUS, GET_SYS, GET_MODULES, GET_NAMES, STOP_ALL
  • BLE 통신·파이프라인 : 서비스 UUID 필터, 페어링, 20바이트 청킹, fire-and-forget 무응답 vs 응답 대기, FIRE_AND_FORGET 양쪽 동기화
  • BATCH 블록 : BATCH;a|b|c 직렬화, 순차 실행·단일 응답, 금지 블록 이중 방어(값반환·제어흐름·중첩)
  • 블록 코딩 엔진 : generateCommand 블록-명령 매핑, 반복/조건/변수/함수, 예제 XML 로드, 한글 로마자 변환
  • 액추에이터 : 서보 휠(연속회전), DC 모터(dual PWM 대칭), LED 6개, 부저(논블로킹), 발사기
  • 디스플레이 : OLED 128x64 MSG/MSG_XY/ICON/CLEAR/CLEAR_RECT
  • 센서 : 초음파 거리(HC-SR04), 디지털 자기장(홀/리드)
  • 설정·안정성 : SYS_SET 영구 저장, 모듈 활성화, 핀 변경, 부팅 None 가드, XML 저장/불러오기
  • 3D 시뮬레이터 : 4개 주제(알비·우주 신호등·발사대·로버), simSink 명령 매핑, 눈/가슴 LED·레이더·로켓·서보 이동·OLED 에뮬, 실행/중지, hold 계산, BATCH 시뮬 (총 24 케이스)
  • E2E : XML 프리셋 실행(Test All Hardware / Launch Rocket), 시뮬레이터 무하드웨어 실행

4. 테스트 전략 (Test Strategy)

  • 단위 테스트 : Pico 핸들러 함수별, Web 모듈별 독립 검증 (예 : ai_helper_grammar.test.mjs)
  • 통합 테스트 : Web → BLE → Pico 데이터 흐름, 명령-응답 정합성
  • 시스템 테스트 : TCL v1.0 98 케이스 전수 종단(E2E) 검증
  • 시뮬레이터 선행 테스트 : 하드웨어 없이 Web/Sim_Parts로 블록-명령 경로를 먼저 검증한 뒤 실기 확인
  • 프리셋 기반 회귀 : Test All Hardware.xml로 전 모듈을 한 번에 구동해 매 빌드마다 하드웨어 회귀 확인

11. 리스크 및 대응 (Risks & Mitigation)

[리스크 1] 교실 BLE 주파수 간섭
· 설명 : 다수 학생이 동시 페어링 시 신호 혼선
· 대응 : 기기 이름(device_name) 구분, 분단별 순차 페어링 안내
[리스크 2] 브라우저 호환성
· 설명 : Firefox/Safari에서 Web Bluetooth 미지원
· 대응 : 진입 시 브라우저 체크 + Chrome/Edge 유도
[리스크 3] 부팅 단계 모듈 초기화 실패
· 설명 : 특정 모듈 미장착 시 초기화 예외로 BLE 응답 불능 우려
· 대응 : hardware.py의 모듈별 try/except + main.py의 if robot.module is not None 가드 (TC-SYS-005, TC-OLED-006, TC-SEN-005로 검증), 단 GUN_FIRE는 예외적으로 가드 없이 gun=None이어도 항상 1을 응답하므로 하드웨어 유무 진단에는 사용 불가
[리스크 4] BATCH 누적 SLEEP > RESPONSE_TIMEOUT(5초)
· 설명 : BATCH 안 SLEEP 합이 5초를 넘으면 Web이 timeout으로 오인
· 대응 : BATCH 안 SLEEP 누적 4초 이내 권장, 긴 대기는 BATCH 밖 SLEEP 블록 사용
[리스크 5] UART 9600 baud 처리량 천장
· 설명 : BLE 모듈-Pico UART가 9600 baud로 고정되어 byte당 전송시간이 천장
· 대응 : AT+BAUD 상향 + Pico UART_BAUDRATE 동기화를 후속 개선 과제로 관리
[리스크 6] 하드웨어 배터리 방전 중 오동작
· 설명 : 수업 중 배터리 전압 저하 시 서보·DC 모터 토크 저하 및 불안정 동작 우려, 현재 get_battery_voltage()는 항상 100을 반환하는 미구현 상태라 전압 경고 기능이 없음
· 대응 : 배터리 전압 ADC 측정·경고 기능 추가를 후속 과제로 등록, 단기적으로 수업 전 완충 점검을 강사 매뉴얼에 명시
[리스크 7] BLE 연결 끊김 시 구동 지속 위험
· 설명 : 모터 구동 중 GATT 끊김이 감지되지 않으면 로버가 계속 움직일 수 있음
· 대응 : disconnect 감지 시 state.isExecuting=false로 실행 중단 + 재연결 안내, 물리 STOP_ALL 경로 확보 (TC-BLE-007로 검증)
