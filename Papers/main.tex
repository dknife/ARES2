\begin{abstract}
\noindent
피지컬 컴퓨팅 교육에서 교육용 로봇 플랫폼은 학습자가 알고리즘을 하드웨어로
직접 구현하며 컴퓨팅 사고력을 체득하게 하는 효과적인 도구이다. 그러나 기존
다수의 플랫폼은 전용 애플리케이션이나 PC용 연결 프로그램의 설치를 전제로 하여,
크롬북(Chromebook)\,\textperiodcentered\,웨일북 등 경량 운영체제 기반 기기가
보급된 교육 현장에서 접근성과 수업 준비 효율을 저해한다. 본 연구는 별도의 설치
과정 없이 웹 표준 브라우저만으로 로버를 제어하는 Web Bluetooth 기반 교육 플랫폼
\keyterm{ARES}(Autonomous Rover Exploration System)를 설계\,\textperiodcentered\,구현한다.
구현의 핵심 난점은 사양이 제각각인 저가 HM-10/BT05 \emph{클론} BLE 모듈과
RAM이 제한된 MicroPython 런타임이라는 척박한 토대 위에서 다양한 길이의 명령과
센서 응답을 양방향으로 신뢰성 있게 주고받는 데 있다. 본 논문은 이를 네 가지 설계
과제로 정식화하고, 각 과제에서 순진한 접근이 실패하는 이유와 채택한 해법
\,\textperiodcentered\,근거를 기술한다. 구체적으로 (1)~BLE 20바이트 전송 제약을
극복하는 개행 종결 청크 스트리밍, (2)~클론 모듈의 비결정성에 대응하는 전송 모드
선택과 페이싱, (3)~비동기 응답 경합을 제거하는 프로미스 선설정과
fire-and-forget 분리, (4)~제한 자원 위에서의 청크 재조립과 접두사 기반 디스패치를
다룬다. 아울러 12차시 48미션 화성 탐사 교육과정을 제시하고, 기능 검증 결과와 위
설계 해법의 효과를 정량 검증하기 위한 측정 프로토콜을 제시한다.

\smallskip
\noindent\textbf{주제어:} Web Bluetooth, BLE, 블록 코딩, Google Blockly,
라즈베리파이 피코, 피지컬 컴퓨팅, 무설치 교육 플랫폼
\end{abstract}

\section{서론}
\label{sec:intro}

지능정보사회의 도래와 함께 소프트웨어\,\textperiodcentered\,인공지능 소양 교육의
저변이 확대되면서, 추상적 개념을 물리적 결과로 환원해 보여 주는 피지컬 컴퓨팅
교육의 중요성이 커지고 있다. 하드웨어를 직접 제어하며 알고리즘을 설계하는 과정은
학습자가 순차\,\textperiodcentered\,반복\,\textperiodcentered\,조건 등의 컴퓨팅
사고 요소를 시각적이고 직관적으로 체득하도록 돕는다.

그러나 기존의 다수 교육용 로봇 플랫폼은 전용 모바일 애플리케이션이나 PC용 드라이버
\,\textperiodcentered\,연결 프로그램의 설치를 필수로 요구하는 구조적 한계를 지닌다.
이러한 방식은 학교\,\textperiodcentered\,공공 교육 현장에 널리 보급된 크롬북,
웨일북 등 경량 운영체제 기반 기기에서의 접근성을 저해하며, 임의 소프트웨어 설치가
제한되는 관리형 단말 환경에서는 도입 자체가 어렵다. 또한 수업 시작 전 기기마다
프로그램을 설치\,\textperiodcentered\,설정하는 데 적지 않은 시간과 관리 비용이
소요되어 실제 학습 활동 시간을 잠식한다.

본 연구는 이러한 \keyterm{설치 장벽} 문제를 해소하기 위해, 별도의 설치 과정 없이
웹 표준 브라우저에서 직접 로버를 제어하는 Web Bluetooth 기술 기반의 교육 플랫폼
\keyterm{ARES}(Autonomous Rover Exploration System)를 제안한다. ARES는 초등학생
수준 학습자를 대상으로 하며, Google Blockly 기반 블록 코딩 인터페이스로 작성한
제어 명령을 브라우저가 BLE(Bluetooth Low Energy)를 통해 라즈베리파이 피코 기반
로버 ``알비''로 직접 전송한다. 학습 내용은 화성 탐사라는 단일 서사 아래 12차시
48미션으로 구조화되어, 단계적 난이도로 블록 코딩 역량을 함양하도록 설계되었다.

무설치라는 목표는 단순히 Web Bluetooth API를 호출하는 것만으로 달성되지 않는다.
교육 현장의 단가 제약상 사용하게 되는 저가 HM-10/BT05 \emph{클론} BLE 모듈은 표준
사양과 달리 한 번에 보낼 수 있는 페이로드가 작고 connection interval과 지원 전송
모드가 제품마다 다르며, 메인 컨트롤러인 라즈베리파이 피코의 MicroPython 런타임은
RAM이 극히 제한적이다. 본 연구의 실질적 기여는 이러한 비이상적 토대 위에서 다양한
길이의 명령과 센서 응답을 \keyterm{신뢰성 있게} 주고받는 통신\,\textperiodcentered\,
재조립 계층을 설계한 데 있다. 구체적 기여는 다음과 같다.

\begin{itemize}
  \item \textbf{무설치 웹 제어 구조:} 중계 플러그인이나 로컬 백그라운드 프로그램
        없이, 웹 표준 Web Bluetooth API만으로 BLE 하드웨어를 직접 제어하는
        3계층 아키텍처를 설계하였다.
  \item \textbf{불안정 링크 대응 경량 통신 프로토콜:} BLE의 20바이트 전송 제약과
        클론 모듈의 비결정성, 비동기 응답 경합, 제한 자원이라는 네 가지 설계
        과제를 정식화하고, 개행 종결 청크 스트리밍\,\textperiodcentered\,전송 모드
        선택과 페이싱\,\textperiodcentered\,프로미스 선설정과 fire-and-forget
        \,\textperiodcentered\,접두사 기반 \code{CommandProcessor}로 해결하였다.
  \item \textbf{시나리오 기반 교육과정:} 화성 탐사 서사를 중심으로 한 12차시
        48미션 교육과정을 플랫폼에 내장하여, 하드웨어 구성 요소를 단계적으로
        도입하는 학습 경로를 제공하였다.
\end{itemize}

본 논문의 구성은 다음과 같다. \ref{sec:related}장에서 관련 연구를 검토하고,
\ref{sec:arch}장에서 시스템 아키텍처를 기술한다. \ref{sec:impl}장에서 네 가지
설계 과제와 그 해결을 상세히 설명하고, \ref{sec:eval}장에서 기능 검증과 측정
프로토콜을 제시한다. \ref{sec:conclusion}장에서 결론과 향후 과제를 논의한다.

\section{관련 연구}
\label{sec:related}

\subsection{교육용 블록 코딩 플랫폼}
스크래치(Scratch)\cite{scratch}, 엔트리(Entry)\cite{entry}, Google
Blockly\cite{blockly} 등은 텍스트 문법 학습 부담을 제거한 시각적 블록 구성을 통해
프로그래밍 입문 장벽을 크게 낮춘 대표적 도구이다. 특히 Blockly는 블록 조립 결과를
JavaScript\,\textperiodcentered\,Python 등으로 변환하는 코드 생성기를 제공하여,
웹 애플리케이션에 임베드되는 블록 에디터의 사실상 표준 라이브러리로 자리 잡았다.

그러나 이들 플랫폼이 실제 하드웨어 로봇과 연동될 때에는 공통적인 불편함이 있다.
웹 브라우저의 샌드박스 보안 모델은 기본적으로 임의의 물리 장치 접근을 허용하지
않으므로, 종래에는 시리얼\,\textperiodcentered\,블루투스 포트를 중계하는 별도의
로컬 백그라운드 프로그램(connector/agent)이나 브라우저 확장을 설치해야 했다. 이
중계 계층은 설치\,\textperiodcentered\,권한\,\textperiodcentered\,버전
호환성 문제를 유발하며, 관리형 단말 환경에서 도입을 어렵게 만드는 주된 요인이다.

\subsection{Web Bluetooth 기반 하드웨어 제어}
Web Bluetooth API\cite{webbluetooth}는 브라우저가 중계 프로그램 없이 직접 BLE
기기와 GATT(Generic Attribute Profile) 통신을 수행할 수 있도록 하는 웹 표준
기술이다. 크로미움 계열 브라우저의 지원 확대로, 웹페이지 접속만으로 물리 장치를
제어할 수 있는 기술적 토대가 마련되었다. 다만 Web Bluetooth는 사용자 제스처
기반의 기기 선택, 보안 컨텍스트(HTTPS 또는 \code{localhost}) 요구, 그리고 BLE
링크 계층의 작은 전송 단위라는 제약을 수반한다. 더욱이 교육 현장에서 비용 문제로
채택되는 HM-10/BT05 호환 \emph{클론} 모듈은 표준 동작이 보장되지 않아, 다양한
길이의 제어 명령과 센서 응답을 양방향으로 안정적으로 주고받기 위해서는 이러한
제약과 비결정성을 흡수하는 응용 계층 프로토콜 설계가 반드시 필요하다. 본 연구는
이 지점에서 경량 청크 스트리밍 프로토콜을 설계하여 교육용 로버 제어에 적용한다.

값이나 상태 정보는 동일한 20바이트 청킹 규칙을 따라 역방향으로 전송되어 웹 UI에
시각화된다.

\section{핵심 기술 구현: 설계 과제와 해결}
\label{sec:impl}

ARES 구현의 본질적 난점은 두 가지 척박한 토대 위에서 \emph{다양한 길이의 명령과
센서 응답을 양방향으로 신뢰성 있게} 주고받아야 한다는 데 있다. 첫째 토대는 단가
제약상 채택하게 되는 HM-10/BT05 \emph{클론} BLE 모듈이다. 표준 사양과 달리 한 번에
보낼 수 있는 페이로드가 작고(약 20바이트), connection interval(약 30\,--\,70\,ms)과
지원 write 모드가 제품마다 다르다. 둘째 토대는 RAM이 킬로바이트 단위인 MicroPython
런타임으로, 넉넉한 수신 버퍼나 정규식 기반 파서를 두기 어렵다. 본 장은 이 문제를 네
개의 설계 과제(C1\,--\,C4)로 나누고, 각 과제에서 순진한 접근이 왜 실패하는지와
채택한 해법\,\textperiodcentered\,근거를 기술한다. 표~\ref{tab:challenges}에 이를

\subsection{과제 C1\,\textperiodcentered\,C2: 불안정한 저가 BLE 링크 위의 가변 길이 스트리밍}
\label{sec:ble}

무선 통신 계층은 Web Bluetooth API 위에 구현되었다. 사용자가 UI에서 연결을
요청하면 브라우저의 기기 선택 절차가 활성화되며, HM-10/BT05 계열 모듈이 노출하는
UART 호환 Service UUID(\code{0000ffe0-\dots})로 대상을 필터링하여 GATT 서버 연결을
수립하고, 데이터 송수신용 characteristic(\code{0000ffe1-\dots})을 확보한다. 보안
컨텍스트 제약에 따라 플랫폼은 \code{localhost} 또는 HTTPS에서 동작한다.

\paragraph{문제(C1).}
BLE 링크에서 한 번에 전송 가능한 페이로드는 작다. ARES는 안전한 상호운용을 위해
페이로드를 \keyterm{20바이트}로 제한한다. 그런데 제어 명령은 길이가 가변적이다.
예컨대 시스템 설정 명령 \code{SYS\_SET,80,10,1,ARES-Rover}는 개행을 포함해 약
28바이트로 단일 청크를 초과하며, OLED 메시지 출력이나 다중 명령 일괄 전송(BATCH)은
수십\,--\,수백 바이트에 이른다. 따라서 가변 길이 명령을 작은 청크로 안전하게
분할\,\textperiodcentered\,복원하는 프레이밍 규칙이 필요하다.

\paragraph{설계 대안과 선택.}
프레이밍 방식의 후보와 그 한계를 표~\ref{tab:framing}에 비교하였다. 고정 길이
패딩은 짧은 명령에도 최대 길이를 전송해 대역을 낭비하고, 시작/종료 플래그(S/E)
방식은 프레임마다 오버헤드가 붙을 뿐 아니라 페이로드 안에 플래그 바이트가 나타날 때
escape 처리가 필요하다. MTU 협상으로 큰 페이로드를 한 번에 보내는 방식은 표준
모듈에서는 유효하나 클론 모듈에서 협상이 비표준적이거나 실패한다. ARES는 제어 명령이
본래 텍스트라는 점을 활용하여, 명령 끝에 \keyterm{개행}(\code{\textbackslash n})을
프레임 종결자로 부가하는 방식을 채택하였다. 오버헤드가 1바이트에 불과하고, 텍스트
명령에 자연스러운 경계이며, 수신 측의 재조립 로직(\ref{sec:c4}절)과도 잘 맞는다.

\paragraph{문제(C2)와 해법.}
클론 모듈은 connection interval과 지원 write 모드가 제품마다 달라, 단일한 송신
전략으로는 안정성과 처리량을 동시에 얻기 어렵다. GATT 응답을 매 청크마다 기다리는
\code{writeValueWithResponse}만 사용하면 interval마다 ACK 왕복이 끼어들어 다중 청크
명령의 처리량이 급락하고, 반대로 지연 없이 연속 송신하면 청크가 손실된다. ARES는
응답을 기다리지 않는 \code{writeValueWithoutResponse}를 우선 사용하여 처리량을
높이되, 해당 모드를 지원하지 않는 characteristic에 대해서는
\code{writeValueWithResponse}로 안전하게 대체(fallback)한다. 또한 청크 사이에는
모듈의 connection interval(약 30\,--\,70\,ms)을 넉넉히 넘어서는 100\,ms의 페이싱
지연을 두어 청크 손실을 줄이되, 마지막 청크 직후에는 지연을 두지 않아 단일 청크
명령이 불필요하게 100\,ms를 손해 보지 않게 한다(코드~\ref{lst:chunk}).

\begin{lstlisting}[language=JavaScript, caption={가변 길이 명령의 개행 종결 20바이트 청크 송신 (Web 측, 발췌)}, label=lst:chunk]
const encodedData = encoder.encode(data + '\n');   // newline-terminated frame
for (let i = 0; i < encodedData.length; i += MAX_CHUNK_SIZE) {  // 20
  const chunk = encodedData.slice(i, Math.min(i + MAX_CHUNK_SIZE, encodedData.length));
  if (useWithoutResponse) await characteristic.writeValueWithoutResponse(chunk);
  else                    await characteristic.writeValueWithResponse(chunk);
  if (i + MAX_CHUNK_SIZE < encodedData.length) await delay(CHUNK_DELAY); // 100ms, last chunk skipped
}
\end{lstlisting}

\subsection{과제 C3: 비동기 양방향 통신의 정합성 확보}
\label{sec:c3}

\paragraph{문제.}
센서 측정처럼 응답이 필요한 명령에서, 전형적인 ``송신 완료 후 응답 리스너 등록''
순서는 미묘한 경합(race)을 일으킨다. 펌웨어의 BATCH 처리가 매우 빨라, 마지막 청크의
\code{writeValueWithoutResponse}가 반환된 \emph{직후} 응답 notify가 도착하는데, 이
시점에 아직 대기 핸들러가 설정되지 않았다면 응답이 처리되지 못하고 버려진다. 또한
LED 점멸이나 모터 구동처럼 즉시 완료되는 출력 명령에까지 응답 왕복을 적용하면, 그
응답이 뒤이어 보낸 다른 명령의 응답 슬롯으로 밀려 들어가 매칭이 어긋난다.

\paragraph{해법.}
첫째, 응답 대기 명령은 송신을 시작하기 \emph{이전}에 \code{resolve}\,\textperiodcentered\,
\code{reject}\,\textperiodcentered\,타임아웃(기본 5\,초)을 담은 프로미스를 미리
설정한다. 마지막 청크 송신 직후 도착하는 응답도 이미 준비된 핸들러가 받으므로 경합이
사라지고, 무선 환경에서의 응답 손실은 타임아웃으로 회수한다. 둘째, LED\,\textperiodcentered\,
모터\,\textperiodcentered\,메시지 출력 등 즉시 완료되는 출력 명령군은 응답 왕복을
생략하는 fire-and-forget으로 처리하여(웹과 펌웨어 양측이 동일한 명령 집합을 공유)
매칭 어긋남을 원천 차단하고 처리량을 높인다. 다수 명령을 한 번에 보내야 할 때에는
\code{BATCH;cmd1|cmd2|\dots} 형식으로 묶어 단일 프레임으로 스트리밍하고 응답은 한 번만
회신한다.

\subsection{과제 C4: 제한 자원 위의 청크 재조립과 명령 디스패치}
\label{sec:c4}

로버의 메인 컨트롤러로는 가성비와 처리 성능이 우수한 Raspberry Pi Pico를
채택하였으며, 펌웨어는 MicroPython 환경에서 구동된다. BLE 모듈을 거쳐 유입된
데이터는 Pico의 하드웨어 UART(9600\,baud) 수신 버퍼에 적재된다.

\paragraph{문제.}
명령은 여러 청크로 쪼개져 시차를 두고 도착한다. 도착한 데이터를 즉시 완전한 명령으로
간주하면, 분할되어 절반만 도착한 BATCH 명령의 앞부분만 실행되는 오작동이 발생한다
(실제로 초기 구현에서 관찰된 결함이다). 그렇다고 매 수신마다 고정 시간을 대기하면
단일 청크 명령조차 매번 지연을 떠안는다. 또한 RAM이 제한된 MicroPython에서 큰 버퍼나
정규식 기반 파서를 두기 어렵다.

\paragraph{해법.}
수신 측은 개행 종결 규칙에 맞추어 청크를 재조립한다. 메인 루프의
\code{\_read\_uart\_line}은 수신 버퍼에 데이터를 누적하다가 개행 문자를 발견하는
\emph{즉시} 한 줄을 완성하여 반환한다. 이로써 단일 청크 명령은 불필요한 대기 없이
즉시 처리되고, 다중 청크로 분할되어 도착하는 명령은 수신 타임아웃(500\,ms)까지
폴링하며 개행까지 누적된다. 타임아웃이 발생해도 부분 데이터를 반환하지 않고 버퍼를
유지하여 다음 호출에서 이어받게 함으로써, 잘린 명령을 정상 명령으로 오인하는 문제를
제거한다. 메모리 보호를 위해 수신 버퍼 상한을 512바이트로 두고 초과 시 초기화한다.

재조립된 명령은 \code{CommandProcessor.process()}가 \keyterm{접두사}를 분석하여 해당
하드웨어 구동 객체로 분배한다(코드~\ref{lst:dispatch}). 접두사 기반 분기는 정규식이나
별도 파싱 테이블 없이 결정적이고 빠르게 동작하여 제한 자원 환경에 적합하다.

\subsection{하드웨어 모듈 구성}
\label{sec:hardware}

하드웨어는 \code{hardware.py}의 단일 \code{robot} 싱글톤을 통해 일관되게 접근되며,
각 모듈은 \code{system.json} 설정으로 개별 활성화/비활성화할 수 있다. 모듈 초기화는
try-except로 보호되어, 특정 부품이 장착되지 않은 경우 해당 모듈을 \code{None}으로
두고 명령 처리 시 안전하게 무시한다. 이 패턴은 학습 차시별로 필요한 부품만 장착하는
교육 환경에 적합하다. 구성 모듈은 다음과 같다.

\begin{itemize}
  \item \textbf{서보 휠(wheel):} 방향 전환 주행. 좌/우 캘리브레이션 계수를 지원한다.
  \item \textbf{DC 모터(dcmotor):} PWM 듀티 기반 속도 제어로 정/역회전.
  \item \textbf{5구 LED 배열(leds):} 채널별 PWM 듀티(\code{duty\_u16})로 0.0\,--\,1.0
        밝기 제어, 패턴/스와이프 효과.
  \item \textbf{SSD1306 OLED(oled):} 16자 자동 줄바꿈 메시지 출력.
  \item \textbf{초음파 거리 센서(distance, HC-SR04 계열):} 전방 장애물 거리(cm) 측정.
  \item \textbf{자기 센서(magsensor):} 자석 감지 여부 반환.
  \item \textbf{부저(buzzer):} 주파수\,\textperiodcentered\,지속시간 기반 발음, 멜로디.
\end{itemize}

\subsection{기능 검증}
다음 항목은 구현된 플랫폼에서 동작이 확인된 사항이며, 괄호 안은 해당 동작이 검증하는
설계 과제이다.
\begin{itemize}
  \item 중계 프로그램 설치 없이 크로미움 계열 브라우저(\code{localhost}/HTTPS)에서
        HM-10/BT05 클론 모듈과의 GATT 연결 및 양방향 통신 수립.
  \item 20바이트 청크 스트리밍을 통한 단일/다중 청크 명령 전송과 펌웨어 측 재조립.
        OLED 메시지, LED 패턴, \code{SYS\_SET}, \code{BATCH} 등 20바이트를 초과하는
        명령의 정상 처리(C1\,\textperiodcentered\,C4).
  \item 센서 측정 등 응답 대기 명령에서 응답 경합 없이 값 회신, 출력 명령의
        fire-and-forget 처리(C3).
  \item 표~\ref{tab:protocol}의 명령 집합에 대한 하드웨어 구동(서보/DC 모터, 5구 LED,
        OLED, 부저)과 센서 응답(\code{DISTANCE}$\rightarrow$\code{DIST:},
        \code{MAGNET}$\rightarrow$\code{MAG:}).
  \item 12차시 48미션 교육과정의 블록 코딩 $\rightarrow$ 명령 생성 $\rightarrow$
        로버 구동 경로 동작.
\end{itemize}

\subsection{무설치 접근성}
ARES는 설계상 어떠한 중계 프로그램\,\textperiodcentered\,드라이버\,\textperiodcentered\,
확장도 설치하지 않으며, 브라우저로 페이지에 접속하는 것만으로 즉시 사용 가능하다.
이는 애플리케이션\,\textperiodcentered\,connector 설치와 권한 설정이 단말마다
필요한 종래 방식과 대비되는 구조적 이점이다. 관리형 단말(크롬북\,\textperiodcentered\,
웨일북) 환경에서 임의 설치가 제한되더라도 도입이 가능하다는 점에서 교육 현장 적용성이
높다. 다만 설치 시간 단축 효과의 정량화(교실 단위 준비 시간 비교)는 통제된 현장
연구를 통한 후속 측정이 필요하다.

\section{결론 및 향후 과제}
\label{sec:conclusion}

본 연구는 Web Bluetooth 웹 표준 기술을 기반으로, 중계 프로그램 설치 없이 브라우저만으로
교육용 로버를 직접 제어하는 무설치 플랫폼 ARES를 설계\,\textperiodcentered\,구현하였다.
본 연구의 실질적 기여는 사양이 제각각인 저가 클론 BLE 모듈과 자원이 제한된
MicroPython 런타임이라는 비이상적 토대 위에서 다양한 길이의 명령과 센서 응답을
신뢰성 있게 주고받는 문제를 네 가지 설계 과제로 정식화하고 해결한 데 있다. 즉, BLE
20바이트 제약을 극복하는 개행 종결 청크 스트리밍(C1), 클론 모듈의 비결정성을 흡수하는
전송 모드 선택과 페이싱(C2), 비동기 응답 경합을 제거하는 프로미스 선설정과
fire-and-forget 분리(C3), 그리고 제한 자원 위에서의 안전한 청크 재조립과 접두사
기반 디스패치(C4)를 구현하였다. 또한 화성 탐사 서사를 중심으로 한 12차시 48미션
교육과정을 플랫폼에 내장하여, 경량 운영체제 기반 단말이 보급된 교육 현장의 설치 장벽
문제를 구조적으로 해소하였다.

향후 과제는 다음과 같다. 첫째, Web Bluetooth를 지원하지 않는 브라우저(예: iOS
Safari) 환경에서도 하드웨어 제어가 가능하도록 WebSocket 브리지 등 대체 통신 경로를
연구한다. 둘째, 다수의 로버가 협력 미션을 수행하는 멀티 에이전트 화성 탐사 시나리오와
이를 위한 통신 스케줄링\,\textperiodcentered\,최적화 알고리즘을 연구한다. 셋째,
\ref{sec:eval}장에서 제시한 측정 프로토콜에 따라 통신 신뢰성을 정량 측정하여 네 가지
설계 해법의 효과를 입증하고, 통제된 현장 연구를 통해 설치 시간 단축과 학습 효과를
검증한다.
